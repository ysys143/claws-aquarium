## Nanobot Session & Context Management Analysis

### 1. Session Lifecycle

Sessions are keyed by `channel:chat_id` (e.g., `telegram:123456`, `cli:direct`).

**Creation/Persistence:**
- `SessionManager` (`nanobot/session/manager.py`) stores sessions as JSONL files in `{workspace}/sessions/{safe_key}.jsonl`
- Each file starts with a metadata line (`_type: "metadata"`) containing `created_at`, `updated_at`, `last_consolidated` pointer, then individual message lines
- `get_or_create(key)` loads from disk into an in-memory `_cache: dict[str, Session]` on first access
- After every agent turn, `sessions.save(session)` flushes back to disk

**Restoration:**
- On load, JSONL is parsed line-by-line; metadata line is separated from message lines
- Legacy path migration: old `~/.nanobot/sessions/` files are auto-migrated to the new workspace path via `shutil.move`

**Key class:** `Session` dataclass with `messages: list[dict]`, `last_consolidated: int`, `add_message()`, `get_history(max_messages=500)`, `clear()`

---

### 2. Context Isolation

Nanobot does NOT use isolated sub-contexts per task type. There is a **single session per channel:chat_id pair**. Channel-level isolation is achieved via the session key:
- `telegram:123` — separate from `discord:456`
- `cli:direct` — CLI session
- `cron:{job_id}` — each cron job gets its own ephemeral session
- `heartbeat` — fixed key for heartbeat-triggered execution

The `session_key_override` field on `InboundMessage` allows channels to override the default key (used for thread-scoped sessions in some chat platforms).

---

### 3. Multi-Agent Orchestration (Sub-Agents)

**Single-depth spawning via `spawn` tool:**
- The main agent can call the `spawn` tool (`nanobot/agent/tools/spawn.py`) to create background sub-agents
- `SubagentManager.spawn()` (`nanobot/agent/subagent.py`) creates a fresh `asyncio.Task` per sub-task
- Sub-agents have their **own isolated tool registry** (no `message` tool, no `spawn` tool — prevents recursive spawning)
- Sub-agents get a stripped-down system prompt, max 15 iterations
- On completion, sub-agent results are re-injected into the main agent's message bus as `InboundMessage(channel="system", sender_id="subagent", ...)`, so the main agent processes the result in the original user's session context

**Coordination pattern:**
```
User → main agent → spawn tool → SubagentManager.spawn() → asyncio.Task
                                                              ↓ (on complete)
User ← main agent ← bus.publish_inbound(system message) ← _announce_result()
```

Sub-agents are tracked by session key in `_session_tasks: dict[str, set[str]]`, allowing `/stop` to cancel all sub-agents for a session.

---

### 4. Conversation History Management

**Context window management:**
- `session.get_history(max_messages=self.memory_window)` (default 100) — returns only unconsolidated messages
- Leading non-user messages are dropped to avoid orphaned `tool_result` blocks
- Tool results are truncated to 500 chars before being saved to session

**Consolidation (LLM-driven summarization):**
- Triggered automatically when `unconsolidated >= memory_window` (runs as background `asyncio.Task`)
- Also triggered explicitly by `/new` command
- `MemoryStore.consolidate()` (`nanobot/agent/memory.py`) calls the LLM with a `save_memory` tool:
  - `history_entry`: 2-5 sentence summary appended to `{workspace}/memory/HISTORY.md` (grep-searchable, timestamped)
  - `memory_update`: full updated long-term facts rewritten to `{workspace}/memory/MEMORY.md`
- After consolidation, `session.last_consolidated` is advanced (messages are NOT deleted — append-only design for LLM cache efficiency)
- `get_history()` uses `messages[last_consolidated:]` to serve only unconsolidated messages to the LLM

**Two-layer memory:**
- `MEMORY.md`: long-term facts (always injected into system prompt via `ContextBuilder.build_system_prompt()`)
- `HISTORY.md`: timestamped grep-searchable log of past conversations

---

### 5. State Persistence

| Layer | Storage | Format | Scope |
|-------|---------|--------|-------|
| Session messages | `{workspace}/sessions/*.jsonl` | JSONL | Per channel:chat_id |
| Long-term memory | `{workspace}/memory/MEMORY.md` | Markdown | Global (workspace) |
| History log | `{workspace}/memory/HISTORY.md` | Timestamped markdown | Global (workspace) |
| Cron jobs | `{data_dir}/cron/jobs.json` | JSON | Global |

### 6. Architecture Layers & Context Management

```
ContextBuilder (context.py)
├── build_system_prompt()  ← identity + AGENTS.md/SOUL.md/USER.md + MEMORY.md + always-skills + skills-summary
├── build_messages()       ← [system] + history + [runtime_ctx + user_message]
└── _save_turn()           ← strips runtime context prefix, truncates tool results, saves to Session

AgentLoop (loop.py)
├── _process_message()     ← loads session, builds context, runs loop, saves turn
├── _run_agent_loop()      ← iterates LLM + tool execution (max 40 iterations)
└── _consolidate_memory()  ← delegates to MemoryStore when unconsolidated >= memory_window

SessionManager (session/manager.py)
├── get_or_create()        ← memory cache + JSONL load
└── save()                 ← full JSONL rewrite

SubagentManager (agent/subagent.py)
├── spawn()                ← creates asyncio.Task, tracks by session
└── _run_subagent()        ← isolated tool registry, 15-iter loop, reports via system message bus

MessageBus (bus/queue.py)
└── asyncio.Queue pairs    ← decouples channels from agent (inbound/outbound)
```

**Key design choices:**
- Sessions are strictly append-only (LLM prompt cache efficiency)
- Memory consolidation runs asynchronously — the current turn is never blocked by it
- Sub-agents communicate back via the bus (no direct coupling), enabling clean cancellation
- Runtime context (timestamp, channel, chat_id) is injected per-message but stripped before saving to session history

## IronClaw (nearai/ironclaw) — Session & Context Management Analysis

### Architecture Overview

IronClaw uses a **three-tier isolation model**: Session → Thread → Turn at the conversation layer, and a separate **Job → JobContext** system at the execution layer. These are linked by `conversation_id` on the `JobContext`. Security runs as cross-cutting layers above both.

---

### 1. Session/Thread/Turn Model (`src/agent/session.rs`)

**Session** (`Uuid` keyed, per-user):
```rust
pub struct Session {
    pub id: Uuid,
    pub user_id: String,
    pub active_thread: Option<Uuid>,
    pub threads: HashMap<Uuid, Thread>,
    pub auto_approved_tools: HashSet<String>,  // session-scoped tool approval whitelist
}
```

**Thread** (conversation sequence within session):
```rust
pub struct Thread {
    pub id: Uuid,
    pub session_id: Uuid,
    pub state: ThreadState,  // Idle / Processing / AwaitingApproval / Completed / Interrupted
    pub pending_approval: Option<PendingApproval>,  // freezes thread until user responds
    pub pending_auth: Option<PendingAuth>,           // intercepts next message for credential input
}
```

**Key security boundary — Auth Mode**: When a tool requests credentials, the thread enters `pending_auth` mode. The *next user message is intercepted before the normal pipeline* — no logging, no turn creation, no history — and routed directly to the credential store. Prevents secrets from leaking into conversation logs.

**Pending Approval pattern**: When `pending_approval` is set, the thread stores:
- `context_messages: Vec<ChatMessage>` — full context snapshot at pause point
- `deferred_tool_calls: Vec<ToolCall>` — remaining calls not yet executed

This allows exact resume after user approves/denies, with no state loss.

**Context building** (`Thread::messages()`): rebuilds `Vec<ChatMessage>` from turns on demand — no persistent in-memory message list. Compaction (`src/agent/compaction.rs`) uses three strategies: Summarize+LLM (writes summary to workspace daily log), Truncate (keep N recent turns), MoveToWorkspace.

---

### 2. Session Manager (`src/agent/session_manager.rs`)

```rust
pub struct SessionManager {
    sessions: RwLock<HashMap<String, Arc<Mutex<Session>>>>,
    thread_map: RwLock<HashMap<ThreadKey, Uuid>>,
    undo_managers: RwLock<HashMap<Uuid, Arc<Mutex<UndoManager>>>>,
}

struct ThreadKey {
    user_id: String,
    channel: String,
    external_thread_id: Option<String>,
}
```

**Channel isolation**: Thread identity is keyed by `(user_id, channel, external_thread_id)`. Same user, same external ID, different channel → different thread. This means Telegram threads cannot bleed into CLI threads even with identical IDs.

**TOCTOU prevention**: `resolve_thread` adopts existing UUID-keyed threads only if not already mapped to another channel, using a double-check pattern after write-lock acquisition.

**Lifecycle hooks**: `OnSessionStart` / `OnSessionEnd` fire as background tasks (fire-and-forget) on session creation/prune.

**Idle pruning**: `prune_stale_sessions()` skips contended sessions (`try_lock().ok()?`), cascades cleanup to `thread_map` and `undo_managers`.

---

### 3. Job Context & ContextManager (`src/context/`)

**JobContext** is the execution context (not conversation context):
```rust
pub struct JobContext {
    pub job_id: Uuid,
    pub user_id: String,
    pub state: JobState,
    pub extra_env: Arc<HashMap<String, String>>,  // credentials for child processes
    pub max_tokens: u64,      // per-job LLM budget
    pub repair_attempts: u32, // stuck recovery counter
}
```

**State machine**: `Pending → InProgress → Completed → Submitted → Accepted` with branches to `Stuck → InProgress` (self-repair) and `Failed`/`Cancelled`. Enforced at type level; invalid transitions return `Err`.

**`extra_env`**: Credentials are passed to shell tools via `Arc<HashMap>` on the job context rather than `std::env::set_var` (which is `unsafe` in multithreaded programs). Each tool invocation clones the Arc cheaply.

**ContextManager** uses a write-lock for the entire check+insert to prevent concurrent creates from racing past the `max_jobs` limit (explicit TOCTOU comment in code).

---

### 4. WASM Sandbox Context Isolation (`src/tools/wasm/`)

**Principle**: "compile once at registration, instantiate fresh per execution" — matches NEAR blockchain patterns.

**Per-execution resource limits** (`src/tools/wasm/limits.rs`):
- Memory: 10 MB default (`WasmResourceLimiter` implements wasmtime `ResourceLimiter`)
- Fuel: 10M instructions (CPU metering via wasmtime epoch/fuel)
- Timeout: 60 seconds wall-clock

Each WASM execution gets a fresh `Store` with its own `WasmResourceLimiter` — no cross-execution state leakage.

**Credential injection** (`src/tools/wasm/credential_injector.rs`): WASM modules *never see credential values*. The host intercepts HTTP requests, matches by host pattern (`*.example.com` wildcard support), decrypts from the secrets store, and injects as headers/query params. `SharedCredentialRegistry` uses `std::sync::RwLock` (not async) so sync tool code can query it. Mappings are removed when an extension is deactivated (`remove_mappings_for_secrets`).

---

### 5. Credential Vault & Session Scope (`src/secrets/crypto.rs`)

**AES-256-GCM with HKDF-SHA256 key derivation**:
```
master_key (env) + per-secret random salt → HKDF-SHA256 → derived_key → AES-256-GCM
```
- Nonce is random per encryption (`OsRng`)
- Wire format: `nonce(12) || ciphertext || tag(16)` + separate `salt(32)`
- Master key held as `SecretString` (zeroed on drop via `secrecy` crate)
- Same plaintext → different ciphertext (different salt each time)

Secrets are **session-scoped by `user_id`** — the store's `get_decrypted(user_id, name)` signature enforces this. Credential injection always passes `user_id` explicitly.

---

### 6. Prompt Injection Defense & Context Management (`src/safety/`)

Four-layer `SafetyLayer` wraps all tool output before it reaches the LLM:
1. **Sanitizer** — Aho-Corasick fast pattern matching + regex for injection patterns ("ignore previous instructions", role-switching, etc.)
2. **Validator** — length/encoding/forbidden pattern checks
3. **Policy** — severity-tiered rules (Critical/High/Medium/Low) → Block/Warn/Review/Sanitize
4. **LeakDetector** — scans at two points: tool output → LLM, and LLM response → user

Tool outputs are wrapped in XML:
```xml
<tool_output name="search" sanitized="true">[escaped content]</tool_output>
```

**Shell env scrubbing**: The shell tool removes sensitive env vars before execution, preventing `env`/`printenv`/`$VAR` exfiltration.

---

### 7. Docker Sandbox Proxy (`src/sandbox/proxy/`)

Containers route ALL HTTP/HTTPS through a host-side proxy:
- **Domain allowlist** — `DomainAllowlist` validates before establishing CONNECT tunnels
- **Credential injection at transit** — `CredentialResolver` trait injects auth headers; containers never hold raw credential values
- **Zero-exposure model**: secrets stay encrypted on host, decrypted transiently at proxy, never enter container environment

---

### Key Architectural Insights

| Concern | Mechanism |
|---|---|
| Session isolation | `(user_id, channel, ext_thread_id)` keying |
| Credential security | Encrypted at rest (AES-256-GCM+HKDF), injected at proxy boundary |
| WASM isolation | Fresh `Store` per execution + fuel/memory/timeout limits |
| Auth input isolation | `pending_auth` mode bypasses pipeline entirely |
| Multi-thread safety | `RwLock<HashMap<_, Arc<Mutex<_>>>>` pattern throughout |
| Context overflow | Three compaction strategies (summarize/truncate/workspace) |
| Stuck job recovery | `Stuck → InProgress` state machine transition with attempt counter |
| Prompt injection | 4-layer SafetyLayer + XML wrapping of tool outputs |

The design philosophy is defense-in-depth: each layer (session, job, WASM sandbox, Docker proxy, safety layer) provides independent isolation. Compromise of one layer does not grant access to secrets or cross-user data.
