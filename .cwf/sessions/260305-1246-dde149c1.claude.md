## E. KEY FINDINGS

### Strengths
1. **Atomic writes**: Flash storage-safe with sync + directory fsync
2. **Multi-layer compression**: Soft summarization + emergency truncation
3. **Sophisticated cache invalidation**: Tracks file existence + mtimes, recursive skill trees
4. **Thread-safe**: RWMutex on context builder + session manager
5. **Recent-first memory**: 3-day lookback for daily notes (recency bias)
6. **Skill demand-loading**: Skills listed in summary, full content loaded on-demand via `LoadSkill()`

### Limitations
1. **No active pruning**: Daily notes grow indefinitely
2. **Fixed 3-day window**: Hardcoded, no config option for memory depth
3. **Summary-only compression**: Once summarized, older conversation detail is lost
4. **No persistent long-term notes API**: `WriteLongTerm()` is primitive (full file overwrite)
5. **Session history loss**: Summarized messages are truncated to last 4, earlier context only in summary
6. **No memory search**: No pattern matching or tag-based retrieval across daily notes

### Architectural Patterns
| Pattern | Implementation |
|---------|-----------------|
| **Write safety** | Atomic temp-file + rename with fsync |
| **Cache invalidation** | Baseline mtime + existence snapshots |
| **Compression strategy** | Background summarization + emergency truncation |
| **Session persistence** | JSON + atomic rename, in-memory cache |
| **Context injection** | Named sections in cached system prompt |
| **Memory window** | Fixed 3-day lookback for daily notes |

## B. Long-Term Memory

### 1. Storage Backend

**Dual-backend**: PostgreSQL (default, `postgres` feature flag) or libSQL/Turso (`libsql` feature flag).

- PostgreSQL uses **pgvector** for vector embeddings + **tsvector/FTS** for full-text search.
- libSQL translates to `F32_BLOB(1536)` + `libsql_vector_idx` for vectors, and **FTS5 virtual table** with sync triggers for full-text.
- Backend selected at compile time via Cargo feature flags + runtime `DATABASE_BACKEND` env var.
- Storage abstracted behind `WorkspaceStorage` enum (`mod.rs:73-79`) and `Database` trait (~60 async methods).

### 2. Data Model

**Two-table relational + vector model:**

```
memory_documents
  id: UUID (PK)
  user_id: STRING
  agent_id: UUID? (multi-agent isolation)
  path: STRING (e.g. "context/vision.md", "daily/2024-01-15.md")
  content: TEXT (full document)
  created_at: TIMESTAMPTZ
  updated_at: TIMESTAMPTZ
  metadata: JSONB

memory_chunks  (child of memory_documents)
  id: UUID (PK)
  document_id: UUID (FK → memory_documents.id)
  chunk_index: INT (0-based position)
  content: TEXT (chunk text)
  embedding: VECTOR(1536)? (NULL until backfilled)
  content_tsv: TSVECTOR (auto-maintained by trigger)
  created_at: TIMESTAMPTZ
```

Key design: documents are stored fully in `memory_documents.content` AND also broken into chunks in `memory_chunks` for search. Search operates on chunks; reads operate on documents. (`document.rs:36-55`, `repository.rs:1-6`)

Path namespace: `user_id + agent_id + path` is the unique key, enabling multi-agent isolation. (`repository.rs:55`)

### 3. Write Path

**LLM tool call triggered.** Write path flows:

1. LLM calls `memory_write` tool with `{content, target, append}` params
2. `MemoryWriteTool.execute()` routes by target:
   - `"memory"` → `workspace.append_memory()` or `workspace.write(MEMORY.md, ...)`
   - `"daily_log"` → `workspace.append_daily_log()` (auto-timestamped `[HH:MM:SS]`)
   - `"heartbeat"` → `workspace.append/write(HEARTBEAT.md, ...)`
   - `"bootstrap"` → `workspace.write(BOOTSTRAP.md, "")` (clear file to end first-run)
   - custom path → `workspace.append/write(path, ...)`
3. Every write triggers **immediate re-indexing**: `reindex_document()` deletes old chunks and inserts new ones with fresh embeddings. (`mod.rs:716-747`)

Rate limit on `memory_write`: **20 calls/200 tokens** per period. (`memory.rs:307`)

**Compaction-driven writes** also occur:
- `ContextCompactor.compact_with_summary()` → generates LLM summary → writes to `daily/{date}.md` with header `## Context Summary (HH:MM UTC)` (`compaction.rs:248-258`)
- `ContextCompactor.compact_to_workspace()` → raw turn dump → writes to `daily/{date}.md` with header `## Archived Context (HH:MM UTC)` (`compaction.rs:262-277`)

**Protected files** (cannot be written via LLM tool): `IDENTITY.md`, `SOUL.md`, `AGENTS.md`, `USER.md`. Any attempt returns `ToolError::NotAuthorized`. (`memory.rs:26-27`, `memory.rs:211-218`)

### 4. Read / Search

Four LLM tools:

| Tool | Method | Description |
|------|--------|-------------|
| `memory_search` | Hybrid (FTS + vector via RRF) | Query-driven; default 5 results, max 20 |
| `memory_read` | Direct path lookup | Full document content by path |
| `memory_tree` | Directory listing | Hierarchical view, configurable depth 1-10 |
| `memory_write` | Write/append | Creates or updates any workspace file |

`memory_search` returns `{content, score, path, document_id, is_hybrid_match}` per result. (`memory.rs:99-111`)

### 5. Indexing — Chunking Strategy

**Word-based sliding window with overlap:**

- Chunk size: **800 words** (≈800 tokens for English)
- Overlap: **15%** (120 words overlap between adjacent chunks)
- Min chunk size: **50 words** (trailing chunks smaller than 50 words are merged with previous)
- Step size: 800 - 120 = **680 words** per step
- Strategy: whitespace tokenization, sliding window

`chunk_document()` is primary; `chunk_by_paragraphs()` is an alternative (unused, marked `#[allow(dead_code)]`) that respects `\n\n` paragraph boundaries. (`chunker.rs:69-113`)

On every write, old chunks are deleted and new chunks are regenerated: full re-index on mutation. (`mod.rs:722-744`)

### 7. Hybrid Search — RRF Algorithm

**Reciprocal Rank Fusion (RRF)** combines FTS and vector results:

```
score(chunk) = sum(1 / (k + rank)) for each retrieval method
```

Default params:
- `rrf_k = 60` (standard RRF constant)
- `pre_fusion_limit = 50` per method (candidates before fusion)
- `limit = 10` (final results returned)
- `min_score = 0.0` (no threshold by default)
- Scores normalized to 0-1 range after fusion (max score = 1.0)

Configurable: `vector_only()`, `fts_only()`, custom `rrf_k`, `min_score`. Hybrid matches (in both FTS and vector results) get additive score boost naturally from the formula. (`search.rs:140-236`)

PostgreSQL implementation:
- FTS: `plainto_tsquery('english', ...)` + `ts_rank_cd` on `content_tsv` tsvector column
- Vector: pgvector cosine distance `<=>` operator, ordered ascending (closer = better)

(`repository.rs:422-504`)

- **Trigger**: runs as a background `tokio::spawn` inside the heartbeat loop (`heartbeat.rs:158-170`)
- **Scope**: only `daily/` directory entries; identity files and MEMORY.md are never touched
- **State**: last-run timestamp persisted to filesystem JSON file, not the DB
- **Best-effort**: hygiene failures are logged but never propagate or crash the agent

**Step 2 — Temporal decay** (`temporal-decay.ts`):
- Formula: `score * exp(-λ * ageInDays)` where `λ = ln(2) / halfLifeDays`
- Default: disabled, `halfLifeDays = 30`
- Date source: 1) filename pattern `memory/YYYY-MM-DD.md` 2) file mtime via `fs.stat()`

**Step 3 — MMR re-ranking** (`mmr.ts`):
- Algorithm: Carbonell & Goldstein MMR (1998)
- Default: disabled, `lambda = 0.7`
- Similarity metric: Jaccard on tokenized content (NOT vector cosine)
- Iteratively selects: `λ * relevance - (1-λ) * max_sim_to_selected`
- Fallback: if `lambda == 1`, pure relevance sort; early exit for ≤1 items

**Step 4 — Score threshold + relaxed keyword floor** (`manager.ts:326–348`):
- If all merged results fall below `minScore` but keyword results exist, relaxes threshold to `min(minScore, textWeight)` for keyword hits only

### B4. Read/Search — Hybrid FTS5 + Vector

**File:** `src/memory/sqlite.rs:287-377`, `src/memory/vector.rs:72-132`

Three-stage recall pipeline:

**Stage 1 — FTS5 BM25 keyword search** (`sqlite.rs:288-328`):
```sql
SELECT m.id, bm25(memories_fts) as score
FROM memories_fts f
JOIN memories m ON m.rowid = f.rowid
WHERE memories_fts MATCH '"word1" OR "word2"'
ORDER BY score LIMIT N*2
```
- Query terms wrapped in quotes to escape FTS5 special chars
- BM25 scores negated (SQLite BM25 returns negative; negated for ranking)

**Stage 2 — Vector cosine similarity** (`sqlite.rs:330-377`):
```rust
// Full table scan on embedding BLOB column
// Computes cosine similarity in Rust (no sqlite-vec extension needed)
let sim = vector::cosine_similarity(query_embedding, &emb);
```
- Optional category + session_id filters to narrow scan scope
- Scores in [0.0, 1.0] range (clamped)
- Zero-similarity results dropped immediately

**Stage 3 — Hybrid merge** (`vector.rs:72-132`):
```rust
final_score = vector_weight * vector_score + keyword_weight * keyword_score
```
- Default weights: `vector_weight=0.7`, `keyword_weight=0.3` (`sqlite.rs:40-46`)
- BM25 scores normalized to [0,1] by dividing by max in result set
- Cosine scores already in [0,1]
- Deduplication by ID — best scores from each source merged
- Fallback: if hybrid returns nothing → LIKE search with up to 8 keywords capped (`MAX_LIKE_KEYWORDS=8`, `sqlite.rs:594`)

**N+1 avoidance:** Full entries fetched in a single `WHERE id IN (...)` query after merge, not N round-trips (`sqlite.rs:534-588`).

### B6. Hybrid Search Algorithm Detail

**File:** `src/memory/vector.rs:57-132`

```rust
pub struct ScoredResult {
    pub id: String,
    pub vector_score: Option<f32>,  // cosine similarity [0,1]
    pub keyword_score: Option<f32>, // normalized BM25 [0,1]
    pub final_score: f32,           // weighted sum
}
```

Algorithm: Linear weighted fusion (NOT RRF, NOT MMR, NOT temporal decay):
1. Collect vector results (pre-scored cosine in [0,1])
2. Normalize keyword scores: `normalized = score / max_kw_score`
3. Final: `vector_weight * vs + keyword_weight * ks` where missing score = 0.0
4. Sort descending by final_score, truncate to limit

No re-ranking, no temporal decay, no MMR diversity enforcement.
