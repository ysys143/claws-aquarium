### 핵심 비전 해석

"10명의 연구원 마인드가 업로드된 하나의 AI 에이전트"

이것은 사실상 3가지를 동시에 요구한다:

1. **지식 통합**: 10명 연구원의 논문, 코드, 노트, 프로젝트 이력을 하나의 검색 가능한 지식 베이스로
2. **업무 자동화**: Notion 프로젝트 관리 + GitHub 코드 관리를 에이전트가 대행
3. **집단 지성 시뮬레이션**: "이 주제에 대해 김 박사님이라면 어떻게 생각할까?"류의 개인화된 응답


## 현실성 평가: 블루투스 샤워기인가?

### 결론부터: 반은 현실, 반은 몽상

파이프라인의 각 조각은 개별적으로 가능하다. 하지만 "10명의 마인드 업로드"를 문자 그대로 해석하면 현재 기술로는 불가능하다. 구분이 중요하다.

### 현실적인 것 (지금 할 수 있는 것)

**(1) Company Knowledge (RAG) -- 난이도: 중**

랩의 논문 PDF + 프로젝트 문서 + 회의록을 벡터 DB에 인덱싱하고, 자연어로 검색하는 것.

- 기술적으로 성숙한 영역. OpenClaw의 메모리 스택(sqlite-vec + LanceDB + BM25 하이브리드 검색)이 이미 이것을 구현하고 있음
- 현실적 규모: 논문 수백~수천 편, 프로젝트 문서 수천 개는 단일 머신에서 처리 가능
- 예상 작업: PDF 파싱 파이프라인 구축 + 임베딩 생성 + 검색 인터페이스
- 기존 도구: LangChain, LlamaIndex, 또는 OpenClaw의 메모리 확장 활용

**(2) Notion 자동화 -- 난이도: 하~중**

Notion API를 통해 페이지 생성/수정, 데이터베이스 쿼리, 캘린더 관리를 에이전트가 수행.

- Notion API는 공식 REST API가 잘 갖춰져 있음
- OpenClaw용 Notion MCP 서버가 이미 커뮤니티에 존재
- 간트차트/타임라인 자동 업데이트, 회의록 정리, 태스크 생성 등은 충분히 가능
- 주의점: Notion의 블록 구조가 복잡해서 "예쁜" 자동 생성은 초기에 시행착오 필요

**(3) GitHub 아카이빙 + 코웍 -- 난이도: 하**

GitHub API + OpenClaw로 코드 정리, PR 리뷰, 이슈 관리 자동화.

- 가장 성숙한 영역. Claude Code/OpenClaw가 이미 GitHub 연동에 최적화
- 20년치 코드 아카이빙: git subtree/submodule 또는 monorepo로 구조화
- 코드 검색, 문서화, PR 자동 리뷰 등은 즉시 가능

### 몽상에 가까운 것 (재정의 필요)

**(4) "10명의 마인드 업로드" -- 현재 불가능**

이것이 의미하는 바에 따라 다르다:

| 해석 | 가능성 | 현실적 대안 |
|------|--------|-------------|
| "김 박사의 사고방식으로 답변해줘" | 불가능 | 김 박사의 논문/노트를 RAG로 검색해서 "김 박사의 연구에 기반하면..." 식으로 답변 |
| "랩 전체의 지식을 통합 검색" | 가능 | Company Knowledge RAG가 이것 |
| "각 연구원의 전문 분야를 구분해서 라우팅" | 부분 가능 | 메타데이터 태깅(저자, 분야, 키워드)으로 검색 필터링 |
| "연구원 간 암묵지까지 포함" | 불가능 | 명시적으로 문서화된 것만 인덱싱 가능 |

핵심 한계: LLM은 "누군가의 마인드"를 시뮬레이션하지 않는다. "누군가가 작성한 문서"를 검색해서 인용할 뿐이다.

**(5) "Prism = 논문 작성 및 협업" -- 확인 필요**

"Prism"이 정확히 어떤 제품인지 불명확:
- tryprism.ai = 보험 BPO 도구 (학술 무관)
- prism-ai.com = 마케팅/디자인 도구
- 학술 논문 공동 집필 도구라면: Overleaf, Typst, Authorea 등이 기존 대안
- AI 논문 작성 보조라면: Elicit, Consensus, Semantic Scholar 등이 기존 대안
- 요청자에게 정확한 URL/제품명 확인 필요


## 현실적 제안: 3단계 접근

### Phase 1: "랩 도서관" (1~2개월, 비개발자도 시작 가능)

목표: 랩의 축적된 지식을 AI로 검색 가능하게

```
[Google NotebookLM] 또는 [OpenClaw + RAG 확장]
    |
    +-- 랩 논문 PDF 업로드 (수십~수백 편)
    +-- 프로젝트 보고서/회의록
    +-- 연구 노트
    |
    --> "우리 랩에서 X 주제 관련 연구한 게 뭐 있지?" 에 답변
```

즉시 시작 가능한 선택지:
- **Google NotebookLM**: 무료, 비개발자 친화적, 50개 소스 업로드 제한
- **OpenClaw + memory extension**: 개발 필요하지만 제한 없음, 하이브리드 검색

Phase 1 산출물: "우리 랩이 아는 것" 검색 엔진

### Phase 2: "자동화된 랩 매니저" (2~4개월, 개발 필요)

목표: Notion + GitHub 업무 자동화

```
[OpenClaw 에이전트]
    |
    +-- Notion MCP 서버
    |     +-- 주간 미팅 의제 자동 생성
    |     +-- 프로젝트 타임라인 업데이트
    |     +-- 논문 진행 상황 대시보드
    |
    +-- GitHub MCP 서버
    |     +-- 코드 리뷰 자동화
    |     +-- 레포 구조화/문서화
    |     +-- 이슈 트래킹
    |
    +-- Phase 1의 RAG 지식 베이스 연동
```

Phase 2 산출물: "랩장님, 이번 주 할 일 정리해줘"에 답변하는 에이전트

### Phase 3: "연구 보조 에이전트" (4~6개월+, 고급)

목표: 논문 검색/분석/집필 보조

```
[연구 에이전트 파이프라인]
    |
    +-- 외부 논문 검색 (arxiv, pubmed, scholar)
    +-- 논문 분석/비교
    +-- 관련 연구 자동 수집
    +-- 초고 작성 보조 (LaTeX/Typst)
    +-- 인용 관리 (Zotero 연동)
```

Phase 3 산출물: "최근 6개월 X 분야 동향 정리해줘"에 답변하는 에이전트


## 에이전트 런타임 선택: 보안을 고려한 재평가

### 보안이 왜 중요한가

연구랩 데이터의 특성상:
- **미발표 논문/특허 데이터** -- 프롬프트 주입으로 유출되면 연구 선점권 상실
- **10명이 공유하는 에이전트** -- 사용자 간 격리 없으면 타인의 미공개 데이터 접근 가능
- **20년치 코드** -- 접근 범위 제어 없으면 전체 노출

### 보안 관점의 프레임워크 비교

8개 Claw 분석(security_report.md) 기준:

| 위협 시나리오 | OpenClaw (보안 Tier 2) | NanoClaw (보안 Tier 2) | IronClaw (보안 Tier 1) |
|--------------|----------------------|---------------------|----------------------|
| 미발표 논문 유출 | 인프로세스 실행, 도구가 파일시스템 직접 접근 | 컨테이너 격리, 에이전트는 호스트 파일 직접 못 봄 | WASM 샌드박스, 비밀은 exists() 체크만 가능 |
| API 키 탈취 | env/stdin 전달, 프롬프트 주입에 노출 | 컨테이너 내부 env, 격리됨 | Credential proxy, 에이전트에게 키 자체가 안 감 |
| 악의적 도구/MCP | 도구 allowlist 있으나 인프로세스 | 도구가 컨테이너 안에서 실행 | WASM fuel/memory limit |
| 멀티유저 격리 | 세션 키 논리적 격리 | 사용자별 컨테이너 물리적 격리 | 보안 계층 기반 격리 |

### 핵심 통찰: 장기기억은 MCP로 분리 가능

OpenClaw의 최대 강점이던 Tier 1 메모리 스택은, 사실 MCP 서버로 외부화할 수 있다:

```
[에이전트 런타임 (어떤 Claw든)] <--MCP--> [Memory MCP Server (sqlite-vec + BM25)]
                                <--MCP--> [Notion MCP Server]
                                <--MCP--> [GitHub MCP Server]
```

이렇게 되면 프레임워크 선택 기준이 바뀐다:

| 기준 | 기억 내장 전제 시 | 기억 MCP 분리 시 |
|------|-----------------|-----------------|
| 내장 메모리 스택 | 높은 가중치 | 낮음 (외부화) |
| 보안/격리 | 보통 | **높음** |
| MCP 지원 | 높음 | 높음 (동일) |
| 코드베이스 크기 | 낮음 | **높음** (비개발팀 유지보수) |

### 수정된 프레임워크 추천

| 순위 | 프레임워크 | 근거 |
|------|-----------|------|
| **1** | **NanoClaw + Memory MCP** | 컨테이너 보안 + 작은 코드("8분이면 이해") + 에이전트 스웜 + 메신저 내장. 기억은 MCP로 보강 |
| **2** | **OpenClaw** | 올인원 편의성 최고. 보안 우려를 감수할 수 있다면 가장 빠른 시작. 커뮤니티/생태계 압도적 |
| **3** | **IronClaw** | 민감 데이터가 극도로 중요하면. 단, Rust + 배포/유지보수 난이도 높음 |

상세 비교:

| | NanoClaw | OpenClaw | IronClaw |
|---|---------|----------|----------|
| 보안/격리 | 컨테이너 격리 | 인프로세스 | WASM 샌드박스 |
| MCP 지원 | O (IPC 브릿지) | O (네이티브) | O (네이티브) |
| 메신저 | Telegram/WhatsApp | Telegram/WhatsApp/Slack | 없음 (별도 구현) |
| 코드 크기 | "8분이면 이해" | ~400K LOC | Rust, 복잡 |
| 멀티유저 | 사용자별 컨테이너 | 세션 키 논리적 | 보안 계층 |
| 에이전트 스웜 | 최초 지원 | 서브에이전트 | 서브에이전트 |
| 배포 | Docker Compose | serverless-openclaw | 수동 |
| 내장 기억 | Tier 3 (CLAUDE.md 위임) | Tier 1 (sqlite-vec+LanceDB) | Tier 1 (pgvector+RRF) |
| +MCP 기억 보강 | Tier 1 수준 도달 가능 | 이미 Tier 1 | 이미 Tier 1 |

NanoClaw가 특히 이 사례에 적합한 이유:
- **컨테이너 = 연구원별 물리적 격리** -- 10명의 데이터가 섞이지 않음
- **에이전트 스웜** -- Phase 3의 검색->분석->집필 파이프라인에 자연스럽게 매핑
- **포크 친화적** -- "포크하고, 커스터마이징하고, 소유하세요"가 공식 철학

### "모델 5.3"에 대해

요청에서 "모델은 5.3"이라 했는데:
- Claude 3.5 Sonnet (claude-3-5-sonnet)을 의미하는 것으로 추정
- 현재 최신은 Claude Sonnet 4.6 (2026년 3월 기준)
- 모델 선택은 비용/성능 트레이드오프. 연구 RAG에는 Sonnet 계열이 가성비 최적
- Opus급은 복잡한 추론(논문 비교 분석, 실험 설계)에만 선택적 사용 권장


## 핵심 조언: "블루투스 샤워기"를 피하려면

### 1. 작게 시작해서 키워라

"10명의 마인드가 담긴 만능 에이전트"를 Day 1에 만들려 하지 마라.
NotebookLM에 논문 50편 올려서 검색해보는 것부터 시작.
이것만으로도 "아, 이게 되네" 하는 순간이 온다.

### 2. 데이터가 없으면 AI도 없다

AI 에이전트의 품질 = 입력 데이터의 품질.
20년치 코드가 정리 안 된 채 하드에 흩어져 있다면, 먼저 정리가 필요.
논문 PDF가 체계적으로 관리되고 있는가? Zotero에 들어있는가?
이 "데이터 정리" 단계가 전체 프로젝트의 50% 이상.

### 3. "마인드 업로드"는 "지식 검색"으로 재정의

"김 박사처럼 생각하는 AI"는 못 만든다.
"김 박사가 쓴 모든 문서를 즉시 참조하는 AI"는 만들 수 있다.
이 차이를 요청자에게 명확히 전달해야 한다.

### 4. 비개발 팀의 현실적 경로

| 경로 | 장점 | 단점 |
|------|------|------|
| NotebookLM부터 시작 | 즉시 시작, 무료, 코딩 불필요 | 50 소스 제한, 커스터마이징 불가 |
| OpenClaw 직접 운영 | 무제한, 완전 커스터마이징 | 서버 운영 + 코드 수정 능력 필요 |
| 외부 개발자에게 위탁 | 빠른 구축 | 비용, 유지보수 의존성 |

가장 현실적 추천: **NotebookLM으로 즉시 시작 + 병렬로 NanoClaw/OpenClaw 파일럿 진행**
