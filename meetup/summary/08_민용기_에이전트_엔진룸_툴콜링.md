# 민용기 - 에이전트의 엔진룸: 도구 호출과 하네스의 모든 것

## 개요

FriendliAI 소속이자 oh-my-opencode 메인테이너인 민용기가 AI 에이전트의 내부 동작 원리인 툴 콜링(Function Calling)을 심층적으로 설명한 발표다. LLM이 텍스트 생성 기반으로 동작하는 구조 위에서 채 템플릿(Chat Template)을 통해 툴 호출이 어떻게 직렬화되고 파싱되는지 원리를 공유하며, 이를 최적화해 모델 성능을 향상시킨 실험 결과와 오픈소스 하네스 프로젝트를 소개했다.

---

## 주요 주제/섹션

### 1. 툴 콜링(Function Calling) 기본 원리 [04:52:50]
- LLM은 텍스트 입력 → 텍스트 출력 구조이므로 외부 세계 상호작용을 위해 Function Calling이라는 특수 기술 필요
- API 요청 시 `tools` 필드에 도구 정의를 포함하면 모델이 어떤 함수를 호출할지 JSON 페이로드로 응답
- 예시: 사용자가 "서울 날씨 어때?"라고 물으면 모델이 `get_weather(city="Seoul")` 호출 → 결과를 다시 모델에 전달 → 그라운딩된 답변 생성

### 2. 채 템플릿(Chat Template)과 내부 직렬화 구조 [04:55:09]
- 트랜스포머 모델은 실제로 텍스트 시퀀스만 처리하므로, JSON 페이로드는 반드시 텍스트로 직렬화되어야 함
- 이 역할을 수행하는 것이 **채 템플릿(Chat Template)**으로, 모델마다 서로 다른 형식의 툴 콜 시그니처 생성
  - 예: GLM 4.7, Gemini, GPT, Claude 모두 내부 포맷이 상이함
- 인퍼런스 엔진(vLLM, SGLang 등)이 모델 출력을 파싱해 개발자에게 깔끔한 JSON 페이로드로 전달

### 3. 채 템플릿 최적화 및 성능 향상 실험 [04:58:02]
- 모델 기본 템플릿이 항상 최적이 아님을 실험으로 확인
- Gemini 2.5 Flash 모델에 자체 툴 콜 파서 + 프롬프트 인젝터를 적용한 결과
  - BFCL(Berkeley Function Calling Leaderboard) 싱글/패럴렐 배치 평가에서 **약 7~8% 성능 향상** 확인
- 해당 도구를 오픈소스로 공개

### 4. 로컬 모델용 하네스 프로젝트(Plugr) [05:00:11]
- 로컬 모델의 제약(제한된 컨텍스트 윈도우, 한국어 미흡)을 하네스 수준에서 보완하는 프로젝트
  - 프롬프트 자동 영어 번역 기능으로 로컬 모델 응답 품질 향상
  - 시스템 프롬프트를 최대한 압축해 컨텍스트 효율 개선
- 파일 편집 시 **해시라인(HashLine) 방식** 도입: 각 라인에 해시 앵커를 부여해 모델이 변경 대상 라인만 지정하도록 유도 → 기존 string replace 방식 대비 효율적

### 5. oh-my-openclaw 프로젝트 소개 [05:00:19]
- oh-my-claudecode → oh-my-codex → oh-my-opencode → **oh-my-openclaw** 시리즈의 네 번째 프로젝트
- 오픈클로의 퍼미션 체계를 기본 해제하고, 멀티 에이전트 오케스트레이션, 메모리 관리, 해시라인 편집 등 다양한 최적화 기술 통합 예정
- 한국에서 오픈클로를 가장 잘 사용하는 개발자들의 워크플로우를 수집해 기능에 반영 중

### 6. Q&A 세션 [05:01:14]
- **로컬 모델 컨텍스트 한계 문제**: 하드웨어(GPU 메모리) 제약이 근본 원인이며, 채 템플릿 수준에서 텍스트 최적화로 일부 보완 가능
- **하네스 시스템 프롬프트 길이 문제**: 클로드코드 같은 도구의 긴 시스템 프롬프트를 압축하는 방향으로 하네스 개발 중
- **안티패턴 vs 좋은 예제 비율**: 발표자 의견은 "좋은 예제를 많이 제공하는 것"이 더 효과적
- **벤치마크 방법론**: BFCL 등 공식 벤치마크 사용, 실제 에이전트 코딩 하네스 환경에서의 터미널 배치 테스트 진행 중
- **TypeScript 선택 이유**: AI SDK(Vercel) 오픈소스 기여에서 시작된 인연; 향후 Rust 전환도 고려 중

## 핵심 포인트 및 세부 내용

- **모델은 텍스트만 본다**: Function Calling은 채 템플릿을 통해 시스템 프롬프트 위에 툴 스키마가 텍스트로 삽입되는 구조이며, 모델은 이를 보고 다음 토큰을 예측하는 방식으로 툴 호출 생성
- **모델별 포맷 차이 인지 필요**: GLM, Gemini, Claude, GPT는 내부 툴 콜 시그니처 형식이 모두 다름 — 동일 모델이라도 채 템플릿 튜닝으로 성능 차이 발생
- **BFCL 7~8% 향상**: 커스텀 프롬프트 인젝터만으로 공식 벤치마크에서 유의미한 수치 개선 달성
- **해시라인 편집**: 기존 old_string/new_string 방식 대신 라인 해시 앵커 기반 편집으로 모델이 변경 범위를 더 정확하게 특정
- **오픈소스 기여 동기**: Vercel AI SDK PR 머지 이후 첫 오픈소스 활동 시작, GitHub 스타 32+ 획득 경험이 현재 활동의 동기
- **oh-my-openclaw**: 발표 시점 기준 공개 1주일 미만의 신규 프로젝트, 한국 오픈클로 최고 사용자들의 워크플로우를 기반으로 기능 구성 중

---

## 결론 및 권장사항

AI 에이전트의 툴 콜링 성능은 모델 자체의 능력뿐 아니라 채 템플릿 설계와 하네스 수준의 프롬프트 최적화에 의해 크게 좌우된다. 개발자들은 API 레이어에만 머물지 말고 직렬화된 텍스트 레이어까지 내려가서 문제를 진단하고, 공식 벤치마크(BFCL 등)를 활용해 개선 효과를 검증하는 접근법을 취할 것을 권장한다.
