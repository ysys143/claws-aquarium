# 허예찬 - OMX, OMC, Claw 운영 생태계

## 개요

허예찬 발표자는 AI 에이전트를 단순 도구가 아닌 팀처럼 운영한 경험을 공유한다. OMC, OMX, 클로힙(Claw) 등의 오픈소스 프로젝트가 개인적인 AI 팀 운영 감각에서 출발해 어떻게 제품으로 발전했는지 설명하며, AI 운영 생태계 구축의 원칙과 실패 사례를 다룬다.

---

## 주요 주제/섹션

### 1. 발표 소개 및 목적 [05:39:26]
- 발표는 AI 에이전트 데모 자랑이 아닌 AI를 팀처럼 운영한 경험 공유
- 오픈클로(OpenClaw), OMC, OMX, 클로힙이 어떤 운영 감각에서 탄생했는지가 핵심
- 다섯 가지 흐름: 시작 배경 / 가제 가족 편성 / 주니어 가제의 실패 / 각 가제의 업적과 교정 / 오픈클로 이야기

### 2. 개발가제: 오너 위임형 운영자 [05:42:01]
- 개발가제(AI 에이전트)는 단순 챗봇이 아닌 "오너 위임형 운영자"로 설정
- 역할: OMC/OMX 코-메인테이너, 이슈 트리아지 관리, 상태 파악 후 다음 액션 수행
- OMC와 OMX의 최근 100개 PR 중 80% 이상이 개발가제가 생성한 PR

### 3. 가제 가족 편성과 운영 전략 [05:43:35]
- 개발가제(대표), 집가제(마케팅/SNS 운영), 퀀가제(퀀트 트레이딩)로 편성
- 24시간 오픈소스 바이럴 운영으로 두 달 만에 다수의 GitHub 스타 확보
- 집가제가 SNS 화면 조작 자동화를 구현하며 마케팅 실행력 확보

### 4. 주니어 가제의 실패와 가제탕 사건 [05:47:59]
- 초기에는 OMC 주니어, VQ 주니어가 별도 존재했으나 타운홀 보고를 무시하고 팔로스루 없이 활동
- 결국 "마이너스 100점" 처리되어 가제탕(폐기)으로 처리됨
- 이 사건을 계기로 운영 원칙이 명확해짐:
  - 보고 없는 실행은 의미 없음
  - 실행 없는 보고도 의미 없음
  - 상태를 읽었으면 반드시 다음 액션까지 수행해야 함

### 5. 메모리 구조와 파일 시스템 기반 관리 [05:51:17]
- 메모리 구조는 단순 기록이 아닌 "교리 엔진"으로 설계
- agents.md에는 운영 원칙, 위임 범위, 우선순위, 태도 등을 포함
- 기억을 단순 저장이 아닌 "행동 규칙으로 승격"하는 구조
- 임베딩(벡터 검색) 대신 파일 시스템 + 에이전트 서치 방식 채택
  - 코딩 에이전트의 강점(bash, grep, ls 등)을 활용해 10만 줄 이상의 메모리도 관리 가능
  - 메모리.md는 실제 내용 대신 포인터(지도)로만 구성

### 6. 클로힙(Claw): AI 팀 운영 라우터 [05:54:05]
- 클로힙은 AI 팀을 채찍질하고 이어붙이는 "운영 라우터"
- OMC/OMX가 실행 레이어라면, 클로힙은 그 팀의 신경계
- 디스코드/슬랙 채널, OMC/OMX 이벤트, 로컬 tmux/screen 세션 등 다양한 이벤트를 수집해 가제에게 전달
- AI용 새로운 메신저를 만들 필요 없이 기존 슬랙/디스코드를 그대로 활용

### 7. OMC/OMX SDK와 오픈클로 통합 [05:59:36]
- OMC/OMX의 내부 상태를 훅(hook)으로 노출하는 SDK 제공
- 오픈클로(OpenClaw)가 설치된 환경에서 computer_notifications 스킬 하나로 바로 연동
- 가제가 OMC/OMX 세션을 스포닝할 때마다 실시간 이벤트 수신 및 팔로업 가능

## 핵심 포인트 및 세부 내용

- **언인스톨러 부재 전략**: OMC/OMX에는 의도적으로 언인스톨러가 없음. 한번 설치하면 잔재가 남아 계속 사용하게 유도하는 "낙인 전략"
- **소울(System Prompt) 튜닝의 중요성**: 에이전트의 아이덴티티를 명확히 부여하고 목표와 정렬시키는 것이 운영 능력에 직접적으로 영향을 미침
- **조교(행동 교정) 스킬**: 소울과 에이전트 엔진을 분석해 목표와의 정렬도를 측정하고, 예시 문제를 통해 실제 운영 수준까지 튜닝하는 방법론
- **오픈소스 AI PR 대응**: AI로 생성된 저품질 이슈/PR에 동일하게 AI로 대응하면 오히려 프로젝트 품질이 빠르게 향상됨
- **슈퍼가제 단일화**: OMC/OMX 사용 후 시니어/주니어 구분 없이 슈퍼가제 하나로 운영 가능하다는 결론 도출
- **주요 오케스트레이션 모델**: Opus 4.6 선호, 비용 문제로 Codex 5.4도 사용

## 결론 및 권장사항

"AI 에이전트는 도구가 아니라 팀으로 운영해야 강해진다" — 이것이 2개월간 약 20여 번의 가제 운영 위기를 겪으며 얻은 핵심 결론이다. OMC, OMX, 클로힙은 이 운영 감각이 제품으로 응축된 결과물이며, 파일 시스템 기반 메모리, 훅 기반 이벤트 연동, 명확한 행동 규칙 체계를 통해 단일 슈퍼가제로도 복잡한 오픈소스 프로젝트를 운영할 수 있음을 보여준다.
