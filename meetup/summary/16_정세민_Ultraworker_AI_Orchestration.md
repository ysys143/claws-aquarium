# 정세민 (Sionic AI) - Ultraworker: AI Orchestration

## 개요

Sionic AI의 정세민은 Claude Code 기반의 AI 에이전트 시스템 "Ultraworker"를 소개한다. 슬랙 연동을 통해 AI가 스펙 작성부터 코드 구현까지 4단계 워크플로를 수행하며, 사람이 각 단계에서 승인·거부를 통해 통제권을 유지하는 구조다. 에이전트의 장기 기억 한계를 보완하기 위해 하이브리드 벡터 검색과 육하원칙 기반 룰베이스 온톨로지를 결합한 메모리 시스템을 자체 구축했다.

---

## 주요 주제/섹션

- **Ultraworker 개발 배경** [07:16:51]
  - 기존 슬랙 기반 Claude/Claudebot 사용 시 슬랙 인테그레이션 버그와 수정 어려움 경험
  - 본질은 "MCP를 붙인 Claude Code"라는 결론 도달
  - Claude Code에 직접 MCP 연동 시 더 안정적이고 잘 동작함을 확인

- **4단계 워크플로 구조** [07:19:01]
  - 1단계: 컨텍스트 탐색 (explore context 스킬)
  - 2단계: 작업 리스트 작성 및 승인 요청
  - 3단계: 테크 스펙 작성 (복잡한 업무에만 적용)
  - 4단계: 구현 완료 후 보고
  - 총 3회 승인 프로세스를 슬랙 블록킷 UI (좋아요 버튼)로 구현

- **슬랙 연동 및 대시보드** [07:25:14]
  - 대시보드에서 진행 중인 쓰레드 및 Claude Code 응답 실시간 관제
  - 잘못된 명령 시 에이전트 강제 종료(terminate) 가능
  - 클로드 코드 세션을 포크(fork)하여 맥락을 이어받아 다음 세션으로 연속 진행
  - 칸반 보드로 작업 상태(진행중/실패/리뷰 필요) 시각화

- **하이브리드 검색 메모리 시스템** [07:23:38]
  - 에이전트 장기 기억 부족 문제 해결을 위해 직접 구축
  - BM25 + 벡터DB(ti/Quadrant) 하이브리드 검색 적용
  - 육하원칙(언제/어디서/누가/왜/어떻게) 기반 룰베이스 온톨로지로 맥락 랭킹 알고리즘 구성
  - 전체 슬랙 기록을 벡터DB에 저장하여 과거 채널 기록 기반 답변 제공

- **질의응답: 파일 시스템 vs. 벡터 검색** [07:31:08]
  - 파일 시스템 검색은 직접적인 키워드 매칭에 의존 → 업무 맥락이 복잡해지면 한계 발생
  - 벡터 검색은 의미적 유사도 탐색 가능하나 키워드가 완전히 다를 경우 한계
  - 그래프DB 도입 이유: 벡터로도 잡히지 않는 관계형 맥락 연결을 위해
  - 메모리 레이어: 파일 시스템 → 벡터DB → 그래프DB 순으로 활용

## 핵심 포인트 및 세부 내용

- **두 가지 핵심 인사이트**
  1. 휴먼-인-더-루프(Human-in-the-Loop): 에이전트가 어떻게 동작하는지 실시간으로 관제·중재하는 것이 가장 중요
  2. 장기 기억 한계: 같은 쓰레드에서도 에이전트가 이전 맥락을 잊고 전혀 다른 방향으로 행동하는 사례 다수 발생

- **퍼스널 토큰 활용**: 에이전트가 발표자 본인 슬랙 ID로 직접 메시지를 보내도록 구성 (봇 토큰/퍼스널 토큰 모두 지원)

- **실제 운용 규모**: 개인 데스크톱부터 RTX 6000 6대, 30대 서버에 설치하여 분산 운용

- **실사용 사례**: 오래된 슬랙 채널 기록을 스캔하여 러스트 서비스 장애 원인(캐시 로즈 이슈, 마이그레이션 이슈 등 3가지)을 자동으로 파악하고 완벽히 답변

- **미래 전망**: 맥락이 지속 축적되고 통합되면 사람보다 맥락을 더 잘 파악하고 업무를 더 잘 수행하는 에이전트가 가능할 것으로 기대

- **커리어 관점**: "2024 AI 프론트 엔지니어 → 2025 바이브 코더 → 2026 AI 마스터" 흐름에 공감하며 실제로 그 방향으로 커리어를 전환 중

## 결론 및 권장사항

AI 에이전트 시스템의 실용성은 단순한 자동화가 아닌, 사람이 실시간으로 관제하고 승인하는 루프 구조와 에이전트의 장기 기억 한계를 보완하는 고도화된 메모리 아키텍처(하이브리드 검색 + 온톨로지)의 결합에서 나온다. 파일 시스템만으로는 복잡한 업무 맥락을 처리하기 어려우며, 벡터DB와 그래프DB를 계층적으로 활용하는 메모리 레이어 설계가 필요하다.
