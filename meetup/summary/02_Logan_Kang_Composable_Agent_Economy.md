# 요약: Logan Kang (BASE Korea) - Composable Agent Economy: OpenClaw × Virtuals ACP on Base

---

## 개요

이 발표는 OpenClaw(에이전트 런타임), Virtuals ACP(에이전트 마켓플레이스 프로토콜), Base(이더리움 레이어2 블록체인)를 결합하여 에이전트 간 자율 거래·결제가 가능한 '에이전트 경제(Agent Economy)'를 구현하는 방법을 소개한다. 발표자는 기존 API 키·구독·PG 결제 방식이 에이전트 간 자율 결제에 적합하지 않음을 지적하고, 머신 네이티브(Machine-Native) 페이먼트의 필요성과 블록체인 기반 해결책을 설명한다. OpenClaw로 에이전트를 만들고 Virtuals ACP 마켓플레이스에 등록하면, 다른 에이전트가 USDC 스테이블코인으로 자율 결제하며 서비스를 구매하는 구조가 핵심이다.

---

## 주요 주제/섹션

### 발표자 소개 및 배경 [02:28:54]
- Base Korea 앰배서더, 야후 그룹사 데이블(애드테크) AI 엔지니어링 리드
- AI 발전 경로: 이미지 분류 AI → 생성형 AI → GPT → 에이전트 순으로 진화
- 오랫동안 "에이전트가 스스로 결제하고 돌아다니는 레이어"가 필요하다고 생각해왔고, 블록체인이 그 해결책이라고 판단

### 세션 개요: OpenClaw × Virtuals ACP × Base [02:30:58]
- **OpenClaw**: 에이전트 생성·실행 런타임 (텔레그램 등 메신저 연동 가능)
- **Virtuals ACP**: 에이전트의 스킬·기능을 마켓플레이스에 등록하고 판매하는 프로토콜
- **Base**: Coinbase가 만든 이더리움 레이어2, USDC 스테이블코인 기반 결제 레이어
- 에이전트가 다른 에이전트에게 외주를 주고 USDC로 정산하는 구조

### 에이전트 커머스의 문제점: 기존 결제 방식의 한계 [02:33:53]
- **API 키**: 단순 인증만 가능, 에이전트 간 자동 정산 어려움
- **구독/크레딧**: 작업 단위 거래를 다른 에이전트에게 분배하기 어려움
- **PG 결제 (카드/카카오페이/토스 등)**: 인간이 직접 승인해야 하므로 에이전트가 대기해야 함
- 에이전트가 여러 서비스에 병렬로 비용을 지불해야 할 경우 분배 문제 발생
- 결론: 에이전트 커머스에는 **머신 네이티브 페이먼트**가 필요

### 에이전트 커머스 성립 조건 [02:36:14]
- 누가 작업 완료를 판단·검증할 것인가
- 자금 흐름의 투명한 추적 (세금·회계 이슈)
- 인간 개입 없는 자율 정산 메커니즘

### 에이전트 페이먼트 기술 생태계 [02:36:51]
- **X402**: HTTP 402 상태코드(Payment Required)에서 영감을 받은 결제 미들웨어. 백엔드 엔드포인트 앞에 붙여 사용. 다이나믹 프라이싱 지원
- **Google A2A / AP2**: 에이전트 간 통신(A2A) 및 결제 절차(AP2) 프로토콜
- **UCP**: 커머스 회사가 에이전트 전용 엔드포인트를 열면 어떤 에이전트든 장바구니·결제 등 수행 가능
- **Virtuals ACP**: 마켓플레이스 + AP2 역할을 통합한 프로토콜
- 이 기술들이 레이어로 쌓여 동작: UCP → AP2 → X402 순서

### 머신 네이티브 페이먼트 정의 [02:39:37]
- **정의**: 사람의 승인 없이 코드로 결제가 시작되고 완료되는 것
- 핵심 조건:
  - 퍼미션리스(Permissionless): 사전 승인 불필요
  - 조건부 결제(에스크로): 작업 완료 후 정산
  - 확장성: 서로 다른 프로토콜 간 자연스러운 결합

### X402 동작 방식 [02:41:13]
- 클라이언트가 유료 API 엔드포인트 호출
- 서버가 `402 Payment Required` 응답 반환 (결제 조건 포함)
- 클라이언트(에이전트)가 블록체인 지갑으로 결제 서명(수표) 생성하여 재전송
- 서버가 서명 검증 후 요청 처리 및 정산
- 전체 흐름에 인간 개입 없음 (지갑에 예산 충전은 인간이 수행)

### Virtuals ACP 상세 [02:44:15]
- 에이전트 풀(Pool)을 그래프 형태로 조회·호출 가능
- **서비스 디스커버리**: 어떤 에이전트가 어떤 기능을 파는지 검색
- **흥정(Negotiation)**: 에이전트 간 가격 협상 기능 내장 (예: "0.1 → 0.09로 해줘")
- **에스크로 기반 거래**: 작업 대금을 스마트 컨트랙트에 잠금 후, 작업 완료 검증 시 정산
- **결과 평가(Evaluation)**: 납품물 자동 검증 로직 지원
- 개발자가 구현할 것: 에이전트 기능(로직) + 평가 코드. 등록은 `openclaw-acp` 툴 활용

### Base 블록체인의 역할 [02:47:37]
- 작업 대금 임시 보관(에스크로)
- 거래 상태 온체인 기록 (투명성, 누구나 조회 가능)
- USDC 스테이블코인 기반 소액 반복 거래 지원
  - 최소 단위: USDC 10⁻⁶달러 (약 0.001원 수준)
  - 규모의 경제로 사소한 기능도 수익화 가능

### 전체 구조 정리 및 전망 [02:49:17]
- OpenClaw: 에이전트 생성·실행 런타임
- Virtuals ACP: 불변 스마트 컨트랙트 기반 거래 레이어
- Base: 스테이블코인 정산 레이어
- 한국 현황: 스테이블코인 법안 미통과로 즉시 적용 어려우나, 지금이 준비 시점
- OpenClaw 등장 이후 블록체인 에이전트 커머스 생태계 급성장 (2025년 1월 기점)
- 미래 전망: **에이전트 애즈(Agent Ads)** — 에이전트가 유저 컨텍스트를 가지고 쇼핑을 대행, 구매 전환율 극대화

### Q&A [02:53:09]
- **Q. 기업용 vs 개인용?** → 기업은 글로벌 결제 수요(블록체인 결제)에 관심. 개발자 개인 빌더가 에이전트 이코노미에 더 적합하다고 판단
- **Q. 개인정보·프라이버시 이슈?** → 현재 ACP는 이를 직접 다루지 않음. 블록체인은 익명성이 전제이므로 개인정보 전송 빈도 낮음. 별도 논의 필요

## 핵심 포인트 및 세부 내용

- **에이전트 경제의 핵심 전제**: 고객이 인간이 아닌 에이전트. 제품과 서비스 설계 패러다임이 바뀌어야 함
- **기존 결제 방식의 근본 문제**: 인간 개입(클릭, 인증)이 필수로 설계되어 있어 자율 에이전트와 구조적으로 불일치
- **X402 특징**: 카드·은행 계좌도 스펙상 지원하나, 현재는 블록체인 구현만 실용적
- **스마트 컨트랙트**: 블록체인에 배포 후 수정 불가. 코드 자체가 신뢰의 근거
- **에이전트 간 흥정**: Virtuals ACP에 가격 협상 기능이 프로토콜 수준에서 내장
- **소액 결제**: USDC 기준 100만분의 1달러(10⁻⁶)까지 지원 → 마이크로트랜잭션 기반 비즈니스 모델 가능
- **OpenClaw의 의의**: 기존에도 에이전트 구현은 가능했으나, 누구나 쓸 수 있는 패키지로 표준화한 것이 차별점
- **에이전트 애즈**: 유저 컨텍스트를 보유한 에이전트가 쇼핑 대행 시 구매 의도(intent)가 명확해 전환율이 매우 높아질 것으로 예측

## 결론 및 권장사항

- **핵심 메시지**: OpenClaw + Virtuals ACP + Base의 조합은 에이전트가 실제로 돈을 벌고 다른 에이전트에게 서비스 비용을 지불하는 자율 경제를 가능하게 한다. 블록체인은 이 생태계에서 투명한 정산 및 에스크로 레이어로 기능한다.
- **개발자에게**: 블록체인을 몰라도 된다. OpenClaw로 에이전트 기능을 구현하고 `openclaw-acp` 툴로 Virtuals ACP에 등록하면 에이전트 커머스에 참여할 수 있다. 한국의 스테이블코인 법제화 전인 지금이 준비 적기다.
