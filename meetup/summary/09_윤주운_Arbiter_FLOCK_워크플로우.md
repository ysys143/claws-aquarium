# 요약: 윤주운 (Sionic AI) - Arbiter: pi-mono 만을 활용하여 워크플로우 자동화 메타 에이전트 만들기

## 개요

Sionic AI의 윤주운 발표자가 홈랩 머신들을 활용한 분산 에이전트 오케스트레이션 시스템 "FLOCK"과, 자연어 명령으로 워크플로우를 자동 생성하는 메타 에이전트 "Arbiter"를 소개한다. Claude Code(오픈 클로)를 코딩 에이전트로 직접 활용하는 대신 장기 컨텍스트·암묵지 관리 레이어로 사용하고, 실제 코딩은 Codex CLI 헤드리스 모드에 위임하는 아키텍처를 구축하였다. 이동 중에도 핸드폰으로 에이전트들을 스티어링하며 생산성을 극대화하는 하이브리드 워크플로우를 목표로 한다.

---

## 주요 주제/섹션

### 문제 인식 및 동기 [05:16:03]
- 맥북은 셀룰러 미지원으로 이동 중 코딩이 불편
- 시도한 방법: iPad + SSH 터미널, 핸드폰 + 채팅 기반 Claude
  - iPad SSH: 한글 입력 불가, 연결 자주 끊김
  - 핸드폰 채팅: OpenCode, Codex 등 코딩 특화 플랫폼 대비 성능 열세
- 목표: 이동 중 핸드폰으로 에이전트를 가이딩하며 작업하는 하이브리드 아키텍처

### FLOCK 시스템 아키텍처 [05:17:26]
- 홈랩 4개 노드로 구성된 분산 에이전트 오케스트레이션 시스템
  - Mac Mini: 오픈 클로 기반 오케스트레이터 노드
  - 워크스테이션 (Thread Ripper + RTX 5090 ×2, RTX Pro 6000 ×1): AI 리서치, 로컬 모델 실험
  - 데스크탑 (RTX 3090 ×2): 보조 GPU 스테이션, 로컬 모델 서빙
  - 시놀로지 NAS: 데이터셋·모델 저장, Mattermost 서버, 10GbE 네트워크 허브 역할
- Tailscale로 전 노드 연결, SSH로 상호 접속

### Mattermost 활용 이유 [05:20:22]
- 로컬 자체 호스팅 슬랙 유사 도구로 프라이버시 보장
- 에이전트를 동적으로 스핀오프: DB 직접 접속해 봇 생성·설정 자동화
- 모든 대화 기록 NAS에 저장 → 로컬 에이전트 파인튜닝 데이터로 활용 가능
- Discord 대비 장점: 보안 이슈(영국 사용자 데이터 열람 정책 논란) 회피

### 로컬 LLM 구성 [05:21:42]
- QwQ 3.2 7B 모델, RTX Pro 6000에서 8비트 양자화로 구동
- 약 50 토큰/초의 준수한 속도
- API 비용 없이 민감 정보·장기 메모리 관리
- 보안상 외부 클라우드에 올리기 어려운 개인·기업 데이터 처리에 활용

### 에이전트 구분: 노드 에이전트 vs 프로젝트 에이전트 [05:22:42]
- **프로젝트 에이전트**: Docker 호환 샌드박스(OrbStack) 안에서 실행
  - 도메인 특화 메모리만 축적
  - 코드 수정 시 직접 작성 금지 → 반드시 Codex CLI 헤드리스 모드에 위임
- **노드 에이전트**: 시스템 어드민 역할, 컨텍스트 오염 없이 인프라 관리에만 집중
  - 보안 화이트리스트 정책 점진적 구축

### 장기 암묵지 관리 레이어 [05:25:00]
- 오픈 클로 내장 메모리 시스템 위에 Items 레이어 추가
- 과거 Claude Code 대화 기록에서 시행착오·노하우 자동 추출 → 임베딩 벡터 저장
- 프로젝트 에이전트에 주입하여 반복 실수 방지
- 코딩 에이전트(Codex)에 컨텍스트를 주입하는 PM 역할 수행

### Arbiter 메타 에이전트 데모 [05:27:16]
- Arbiter: 자연어 명령으로 워크플로우를 자동 생성·실행하는 메타 에이전트
- 데모 시나리오: "opus 4.1 → opus 4.6 모델명 변경" 요청
  - 메모리에서 유사 이슈 자동 검색
  - Codex에 위임하여 코드 수정·PR 생성
  - 결과 리뷰 후 새 이슈 발견 시 아이템 엔트리로 저장
- Coyote 에이전트: PM 역할 + Codex 위임 + 결과 리뷰 담당

### 에이전트 팀 구성 [05:30:15]
- **Node**: 머신 시스템 관리
- **Kestrel**: 헤드리스 Chrome으로 UI 직접 클릭·스크린샷 기반 QA
- **Coyote**: Arbiter 프로젝트 PM, Codex 위임 코딩
- **Raven**: 리서치 담당, arXiv 논문 링크 분석·메모리 저장, VLM 서버 연동 실험 자동화
- **Owl**: 로컬 QwQ 모델로 타운 스퀘어 대화 히스토리 트래킹·리포트 작성

### 서브에이전트 미사용 이유 [05:36:47]
- 오픈 클로의 서브에이전트는 부모 에이전트의 모델·메모리·시스템 프롬프트를 모두 상속
- 도메인별 메모리 격리가 불가 → 컨텍스트 오염 발생
- 대신 독립 에이전트 인스턴스를 Mattermost 채널 통해 연결하는 구조 채택

## 핵심 포인트 및 세부 내용

- **추상화 계층 분리**: 개발자가 Claude를 사용하듯, 프로젝트 에이전트가 Codex를 사용하는 상위 추상화 레이어 구현
- **컨텍스트 오염 방지**: 프로젝트별·노드별 메모리 완전 분리로 에이전트 전문성 유지
- **모니터링 필요성**: Session Send(직통 에이전트 통신)보다 Mattermost 경유를 선호 — 에이전트 간 셀프-리인포싱(서로 동의만 반복) 방지 및 사람 개입 용이
- **로컬 모델 활용 기준**: 코딩·오케스트레이션엔 고성능 클라우드 모델, 민감 데이터·장기 메모리 관리엔 로컬 모델
- **향후 계획**: GPU 디스패칭 큐 및 우선순위 프로토콜 정비, 일일 스탠드업 자동 보고, 에이전트 간 지식 공유 강화

## 결론 및 권장사항

FLOCK은 홈랩 인프라와 오픈 클로를 결합해 "이동 중 핸드폰 스티어링"을 실현하는 개인 맞춤형 분산 에이전트 시스템으로, 핵심 설계 원칙은 에이전트별 메모리 격리와 코딩 작업의 Codex 위임이다. 코딩 에이전트를 직접 사용하기보다 장기 암묵지·컨텍스트를 관리하는 PM 레이어로 Claude를 활용하고, 복잡한 작업일수록 사람이 직접 개입하는 단계적 자동화 전략이 실용적임을 보여준다.
