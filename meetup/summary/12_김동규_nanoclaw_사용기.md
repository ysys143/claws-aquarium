# 김동규 (NomaDamas) - Andrej Karpathy가 추천한 nanoclaw 사용기

## 개요

- 클로드 코드 해커톤 우승자 김동규가 OpenClaw 대신 nanoclaw를 선택하게 된 계기와 사용 경험을 공유한다. nanoclaw는 핵심 엔진이 약 4,000줄로 구성된 경량 자율 에이전트 프레임워크로, Andrej Karpathy가 "AI 에이전트의 머릿속에도 들어올 정도의 코드베이스"라며 칭찬한 프로젝트다. 발표자는 이를 포크해 Slack 연동, 크론잡 최적화, 공유 지식 저장소 등 자신만의 커스텀 에이전트 시스템을 구축한 경험을 소개한다.

---

## 주요 주제/섹션

### 발표자 소개 및 배경 [06:14:03]
- 클로드 코드 해커톤에서 팀과 함께 총상금 1만 달러 크레딧 수상, 5,000달러씩 분배
- 수상 크레딧은 Opus 모델 집중 사용으로 발표 시점 기준 500달러 이하로 소진
- autorag(GitHub 스타 4.6K), 신논현 근처 AI 오픈소스 해커하우스(NomaDamas) 운영
- Claude Code + Codex를 활용해 10개 이상의 오픈소스 프로젝트를 동시 운영 중

### OpenClaw 사용 실패 경험 [06:16:56]
- WSL2(게이밍 PC) 환경에서 Slack 연동 시도 → 원인 불명의 연동 실패
- Linux 서버 설치 시 사용자 환경 꼬임으로 스킬/익스텐션 설치 불가
- 세 번째 시도(1080Ti 2장, RAM 32GB 데스크탑)에서 OpenClaw 실행 중 컴퓨터 다운
- 세 번의 실패 후 대안 탐색 → Andrej Karpathy가 칭찬한 nanoclaw 발견

### nanoclaw 선택 이유 [06:19:26]
- 핵심 코드베이스 약 4,000줄: 인간과 AI 에이전트 모두 이해 가능한 규모
- 모든 실행이 컨테이너 기반: 보안성 확보, 자율성과 보안의 트레이드오프를 직접 조율 가능
- 설정 방식: 외부 PR 미수용, "스킬" 단위로만 인티그레이션 추가 (Telegram 스킬, Slack 스킬 등)
  - 기성 인티그레이션 대신 Claude Code / Codex에 인스트럭션을 던져 직접 구현
- "포크가 곧 제품"이라는 컨셉: 사용자가 코드를 직접 수정해 자신에게 맞게 커스텀

### nanoclaw 아키텍처 상세 [06:22:26]
- 단일 Node.js 프로세스 + SQLite 구성
- 메시지 수신 시마다 Docker 컨테이너를 스폰하는 방식 (유휴 시 자동 정리)
- Slack 소켓 모드(WebSocket)로 상시 연동
- 에이전트 간 통신: 마운트된 JSON 파일로 메시지 전달 → 5초 주기로 Node.js 프로세스가 파일 변경 감지 → DB 저장 후 컨테이너 라우팅
- `.env` 등 민감 파일은 자동 차단 패턴으로 에이전트 접근 원천 차단

### Slack 스레드 컨텍스트 설계 [06:26:06]
- 채널 전용 에이전트: 해당 채널 전체 메시지를 컨텍스트로 수신
- 다른 채널에서 호출된 스레드: 해당 스레드 메시지만 격리된 새 세션으로 처리
  - 예) 개발팀 채널에서 영업 에이전트 호출 시, 개발팀 전체 컨텍스트 불필요
- 멘셔닝으로 다른 에이전트 스폰, 1:1 채널에서는 에이전트가 자동 응답

### 크론잡 최적화 [06:28:52]
- 기존 방식: 주기마다 에이전트를 완전히 깨워 작업 수행 → 토큰 낭비
- 개선 방식: 2단계 레이어
  1. JavaScript/Bash 코드 스니펫으로 조건 확인 (예: 새 GitHub 이슈 존재 여부)
  2. 조건이 false면 에이전트 호출 없이 종료, true일 때만 에이전트 스폰
- 15분 주기 GitHub 이슈 체크 등에 적용해 토큰 절약

### 실제 활용 사례 [06:30:04]
- **도비**: 메인 오케스트레이터 에이전트
- **홍명보**: 오픈소스 프로젝트 홍보 마케터 에이전트 (Reddit 댓글, SNS 트렌드 모니터링)
- **투두몬**: 태스크 관리 에이전트
- **연구(autorag PM 에이전트)**: GitHub 이벤트 구독 → 이슈 발생 시 Codex 호출해 PR 자동 생성, HuggingFace/arXiv 최신 논문 자동 탐색 후 이슈 등록 및 구현 PR 제출
- **세컨드 브레인**: Obsidian 기반 Markdown 지식 저장소, 정리 에이전트가 인박스 자료 자동 분류/태깅, Git으로 버전 관리 및 전 디바이스 공유, 모든 에이전트가 공유 지식 베이스로 활용
- **노이즈 캔슬**: SNS 정보를 AI가 분류해 유용한 정보만 세컨드 브레인으로 입력

## 핵심 포인트 및 세부 내용

- nanoclaw는 OpenClaw(약 40만 줄)와 달리 ~4,000줄로 구성되어 에이전트와 개발자 모두 전체 코드를 머릿속에 담을 수 있는 수준
- Andrej Karpathy(OpenAI 공동창업자, 바이브코딩 용어 창시자)가 트위터에서 nanoclaw를 직접 칭찬
- OpenClaw의 대규모 바이브코딩 방식 개발이 초기 불안정성의 원인이라고 분석
- "포크가 제품"이라는 컨셉은 에이전트 네이티브 오픈소스의 새로운 배포 포맷이 될 가능성
- 컨테이너 격리로 보안성 확보하면서도 코드베이스가 작아 자율성 범위를 직접 통제 가능
- 에이전트 간 JSON 파일 기반 비동기 통신 패턴은 단순하고 디버깅 용이
- 크론잡의 토큰 낭비 문제를 2단계 레이어(경량 사전 체크 → 조건부 에이전트 호출)로 해결

## 결론 및 권장사항

- **nanoclaw 적합 대상**: 커스터마이징을 즐기는 개발자, 기성품보다 직접 만드는 것을 선호하는 사람, 코드베이스 전체를 이해하며 제어하고 싶은 개발자
- **일반 사용자에게는 OpenClaw 추천**: 안정성과 기능 완성도 면에서 대부분의 사용 케이스를 충족
- **설계 전 명세 구체화가 핵심**: 무엇을 어떻게 만들지 명확히 하지 않으면 에이전트가 예상치 못한 방식으로 동작하며 디버깅에 큰 비용이 발생
- 코드베이스를 작게 유지하고 인티그레이션은 스킬로 분리하는 방식이 에이전트 네이티브 오픈소스의 유망한 포맷
