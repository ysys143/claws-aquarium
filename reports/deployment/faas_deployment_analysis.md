## Executive Summary

Claw 기반 AI 에이전트의 클라우드 배포는 **워크로드 특성에 따라 완전히 다른 최적 솔루션**을 요구합니다.

### 핵심 발견

| 요소 | 최고 성능 | 최저 비용 | 최고 보안 | 최적 Claw 적합 |
|------|---------|---------|---------|---------------|
| **콜드스타트** | Cloudflare Workers (50ms) | Cloudflare Workers | AWS Lambda (온디맨드) | — |
| **월 비용 (무료)** | Cloudflare Workers | Cloudflare Workers FREE | Google Cloud Functions | AWS Lambda Free Tier |
| **실행 시간당** | Google Cloud Functions | Cloudflare Workers | AWS Lambda (KMS 지원) | NanoBot 호스팅 추천 |
| **24/7 가능성** | 불가능 (모든 FaaS) | 컨테이너 또는 VPS | 컨테이너 또는 VPS | [컨테이너 추천] |
| **에이전트 자율성** | AWS Lambda (IAM) | Google Cloud Functions | IronClaw Docker 기반 | [헤드리스 최적] |

## 아키텍처 비교

### 1. AWS Lambda

```
┌─────────────────────────────────────┐
│       Lambda Invocation             │
│  (Synchronous / Asynchronous)       │
└────────────┬────────────────────────┘
             │
     ┌───────┴───────┐
     ▼               ▼
┌─────────┐    ┌──────────┐
│ Warm    │    │ Cold     │
│ Start   │    │ Start    │
│ 1-5ms   │    │ 200-800ms│
└────┬────┘    └─────┬────┘
     │               │
     └───────┬───────┘
             ▼
    ┌─────────────────┐
    │ Execution Layer │
    │ • Node.js 22.x  │
    │ • Python 3.13   │
    │ • Go 1.22       │
    │ • Java 21       │
    │ • .NET 8        │
    └────────┬────────┘
             │
    ┌────────┴────────────────────┐
    ▼                             ▼
┌──────────────┐        ┌──────────────────┐
│ VPC (옵션)   │        │ IAM + KMS 암호화 │
│ • ENI 할당   │        │ • 환경변수 암호화 │
│ • 냉각 5초   │        │ • Secrets Manager│
└──────────────┘        └──────────────────┘
```

**특징**:
- **격리 모델**: 각 호출마다 새 MicroVM 할당 (Firecracker 기반)
- **메모리**: 128MB ~ 10,240MB (1MB 단위)
- **타임아웃**: 최대 15분 (900초)
- **동시성**: Reserved/Provisioned/Unreserved 3가지
- **저장소**: /tmp 512MB (임시, 휘발)

**보안**:
- IAM 역할별 세분화된 권한
- KMS 환경변수 암호화
- VPC 마운트로 DB/캐시 격리 가능
- [주의] UID 0 실행 (root) 가능 <- 보안 위험

### 2. Cloudflare Workers

```
┌──────────────────────────────────────┐
│   Global Request @ Edge              │
│   (200+ 데이터센터)                  │
└────────────┬─────────────────────────┘
             │
    ┌────────┴──────────┐
    ▼                   ▼
┌──────────────┐  ┌──────────────┐
│ V8 Engine    │  │ Service Worker
│ Snapshot     │  │ API
│ 10-50ms      │  │
└──────┬───────┘  └─────┬────────┘
       │                │
       └────────┬───────┘
                ▼
        ┌────────────────────┐
        │ Isolation Model    │
        │ • 메모리: 128MB    │
        │ • CPU: 50ms/30s    │
        │ • 디스크: 없음     │
        └─────────┬──────────┘
                  │
    ┌─────────────┴─────────────┐
    ▼                           ▼
┌──────────────────┐  ┌──────────────────┐
│ Durable Objects  │  │ KV Store + R2    │
│ (상태 유지)      │  │ (글로벌 캐시)    │
│ 0.6ms 응답       │  │                  │
└──────────────────┘  └──────────────────┘
```

**특징**:
- **격리**: V8 Context Isolation (스냅샷 기반, 공유 메모리 아님)
- **실행 시간**: 초당 최대 30초 (추가 비용으로 확장 가능)
- **콜드스타트**: 거의 없음 (V8 스냅샷 기반)
- **상태 유지**: Durable Objects (강한 일관성, 0.6ms RTT)
- **저장소**: KV (약한 일관성), R2 (S3 호환)

**보안**:
- JavaScript 샌드박스 (메모리 격리)
- 환경변수 노출 제한 (Secrets 사용 권고)
- [장점] 파일시스템 없음 -> RCE 제한적
- [주의] 네트워크 전체 차단 불가 (HTTP 필수)

### 3. Google Cloud Functions

```
┌────────────────────────────────────┐
│  HTTP / Pub/Sub / Cloud Events     │
└────────────┬───────────────────────┘
             │
    ┌────────┴────────┐
    ▼                 ▼
┌─────────┐      ┌──────────┐
│ 2nd Gen │      │ 1st Gen  │
│ (추천)  │      │ (구식)   │
│ 80-400ms│      │ 200-800ms│
└────┬────┘      └──────┬───┘
     │                  │
     └────────┬─────────┘
              ▼
    ┌────────────────────┐
    │ Execution Runtime  │
    │ • Node.js 22.x     │
    │ • Python 3.13      │
    │ • Go 1.22          │
    │ • Java 17          │
    └──────────┬─────────┘
               │
    ┌──────────┴──────────────┐
    ▼                         ▼
┌──────────────┐    ┌──────────────────┐
│ 공유 테넌트  │    │ Cloud Logging    │
│ • 메모리공유 │    │ + Monitoring     │
│ • 포트 무선 │    │                  │
└──────────────┘    └──────────────────┘
```

**특징**:
- **격리**: 테넌트 간 공유 가능 (비용 최적화)
- **2nd Gen**: 더 빠른 부팅, 더 나은 가격
- **메모리**: 256MB ~ 16GB (256MB 단위)
- **타임아웃**: 최대 60분 (단, 콘텍스트 유지 비용)
- **확장성**: 초당 최대 1,000 동시

**보안**:
- Cloud IAM + Service Account
- Secret Manager (환경변수 동적 로드)
- [주의] 테넌트 격리 약함 <- 멀티테넌트 환경
- Cloud Audit Logs 통합

### 4. Azure Functions

```
┌──────────────────────────────────┐
│ Trigger (HTTP / Timer / Queue)  │
└────────────┬─────────────────────┘
             │
    ┌────────┴──────────┐
    ▼                   ▼
┌────────────────┐  ┌────────────┐
│ Consumption    │  │ Premium /  │
│ Plan (서버리스)│  │ Dedicated  │
│ 200-800ms      │  │ 50ms       │
└────────────────┘  └────────────┘
             │
    ┌────────┴──────────────────┐
    ▼                           ▼
┌──────────────┐       ┌──────────────────┐
│ .NET / Node  │       │ Python / Java    │
│ 가장 최적화  │       │ 느린 부팅        │
└──────────────┘       └──────────────────┘
             │
    ┌────────┴──────────────────┐
    ▼                           ▼
┌──────────────┐      ┌──────────────────┐
│ Key Vault    │      │ Managed Identity │
│ (암호화)     │      │ (RBAC)           │
└──────────────┘      └──────────────────┘
```

**특징**:
- **격리**: Consumption Plan (공유 인프라), Premium/Dedicated (격리)
- **메모리**: Consumption 자동, Premium 선택 (1.75GB ~ 14GB)
- **타임아웃**: Consumption 5분, Premium 30분
- **언어**: C#/F# (최고 성능), Node/Python (느림)

**보안**:
- Azure Key Vault (환경변수 암호화)
- Managed Identity (AD 통합)
- VNet 통합 (Premium+)
- [주의] Consumption은 다인 공유 실행 파일 <- 보안 위험

## 24/7 운영 가능성

### FaaS의 본질적 한계

**모든 FaaS는 24/7 상시 구동에 부적합**하다. 그 이유:

1. **이벤트 기반 아키텍처**: 요청 없으면 차단 상태
2. **타임아웃 제한**: Lambda (15분), Azure (5-30분), GCF (60분)
3. **웹소켓 불지원**: 장시간 연결 불가 (Lambda, GCF는 기본 지원 안 함)
4. **상태 비저장**: 메모리 상태는 다음 호출마다 소실

### 24/7 운영을 위한 대안

#### 옵션 1: "Heartbeat + Pull" 패턴 (비효율적)

```
┌─────────────┐
│ CloudWatch  │
│ Event       │
│ (분 단위)   │
└──────┬──────┘
       │
       ▼
┌─────────────────────┐
│ Lambda 함수         │
│ (상태 확인)         │
│ -> SQS 폴링         │
│ -> 작업 수행        │
└─────────────────────┘
       │
       ▼
┌─────────────────────┐
│ 외부 상태 저장      │
│ • DynamoDB          │
│ • 또는 Durable Obj  │
```

**문제점**:
- Lambda는 스테이트리스 -> 상태 추적 어려움
- 외부 API 호출만 가능, 내부 자율 작업 제한적
- "보내기 -> 기다리기 -> 응답받기" 체인 구성 복잡

#### Cloudflare Durable Objects + 권한 위임

```javascript
// Durable Object 내 자율 작업
export class Agent {
  async autonomous_task() {
    // 메모리상 상태 유지
    this.state = await this.storage.get("agent_state");

    // 자율적 판단
    const decision = await this.llm.decide(this.state);

    // 즉시 실행 (웹소켓 유지)
    await this.api.execute(decision);

    // 상태 저장
    await this.storage.put("agent_state", this.state);
  }
}
```

**장점**:
- 강한 일관성 보장
- 상태 자동 보존
- 0.6ms 응답

**단점**:
- 비용 누적 ($1.25/GB-일)
- 영구 상시 구동 시 가격 폭증

### MQTT 로컬↔클라우드 분리 패턴

**핵심 인사이트**: 로컬 머신에 인바운드 포트를 열지 않고도 클라우드와 실시간 양방향 통신 가능.

```
[텔레그램/외부 서비스]
        ↓ (HTTPS only)
[AWS Lambda 에이전트 루프]
        ↓ (MQTT WebSocket)
[로컬 데스크탑 앱]
 - 아웃바운드 연결만 유지
 - 퍼블릭 IP 불필요
 - 포트 포워딩 불필요
 - Tailscale 불필요
```

**기존 로컬 에이전트 문제점 vs MQTT 해결책**:

| 문제 | 기존 방식 | MQTT 해결 |
|------|---------|----------|
| 외부 접근 | 포트 포워딩 필요 | 아웃바운드만 |
| 보안 | 퍼블릭 IP 노출 | 로컬 머신 비노출 |
| 확장성 | 머신 추가 구매 | 인스턴스 복제만 |
| 백업 | 머신 죽으면 로그 소실 | 외부 저장소 분리 |

### TypeScript + tsc 타입체크 루프 — Tool-Call 패턴

**소스**: 이상현 발표

**패턴 설명**: LLM이 도구(Tool)를 별도 정의하는 대신, TypeScript 코드를 직접 작성 → tsc 타입 체크 → 통과 시 실행하는 방식.

```
[LLM]
  → TypeScript 코드 생성
  → tsc 컴파일 실행
  → 타입 오류 시: LLM이 재작성 (자동 루프)
  → 통과 시: 코드 실행
  → 결과를 LLM에 반환
```

**장점**:
- 별도 툴 정의(Tool Schema) 없이 타입 정의 파일만으로 안전한 도구 호출 구현
- Google Calendar, Gmail 등 REST API는 TypeScript 타입 정의만 작성하면 바로 호출 가능
- MCP나 CLI 없이도 API 직접 호출로 충분

**비교**:
- 기존 MCP 방식: 도구를 JSON Schema로 정의 → LLM이 structured output으로 호출
- TypeScript 루프 방식: 코드 자체가 도구 — 타입 안전성이 검증 메커니즘
