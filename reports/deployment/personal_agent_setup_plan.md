## 1. 내 계획 — 원하는 시스템

### 1.1 시스템의 핵심 아이디어

내가 원하는 것은 단순한 챗봇이 아니다. 에이전트가 먼저 나에게 말을 거는 시스템이다. 보통 AI 도구는 사용자가 질문하면 답하는 방식으로 작동한다. 내가 구상하는 시스템은 반대 방향이다. 모니터링 에이전트가 24시간 개발자 커뮤니티를 순회하다가 흥미로운 것을 발견하면, 그것이 Telegram 메시지로 나에게 먼저 날아온다. 나는 그 알림을 보고 "이거 실험해봐"라고 답하거나, 무시하거나, 더 물어보거나 한다.

이 "아웃바운드(outbound) HITL(Human-in-the-Loop)" 패턴이 시스템 전체의 핵심이다. 에이전트가 능동적으로 정보를 수집하고 판단하며, 사람의 개입이 필요한 시점에만 Telegram으로 연락한다. 나는 항상 제어권을 갖고 있지만, 에이전트가 먼저 움직인다.

### 1.2 전체 워크플로우

시스템이 어떻게 돌아가는지 단계적으로 서술하면 다음과 같다.

**1단계 — 모니터링과 탐지**

모니터링 에이전트는 별도의 세션에서 6시간마다 깨어난다. Reddit의 r/MachineLearning, r/LocalLLaMA, HackerNews, arXiv의 cs.AI/cs.CV 섹션, 주요 Discord 서버, X(구 Twitter)의 기술 계정들을 순회한다. 단순히 크롤링하는 것이 아니라 LLM을 사용해 "이것이 실험해볼 만한 가치가 있는가"를 판단한다. 예컨대 새 OCR 라이브러리가 state-of-the-art를 주장하며 GitHub에 올라왔다면, 에이전트는 그 레포의 README를 읽고, 기존 도구들과 비교해, 실험 가치가 있다고 판단하면 나에게 Telegram 메시지를 보낸다.

이것이 **첫 번째 HITL 체크포인트**다. 에이전트는 멈추고 나의 결정을 기다린다.

**2단계 — 연구 실험**

내가 "실험해봐"라고 답하면, 오케스트레이터가 연구 에이전트를 별도 세션으로 스폰한다. 연구 에이전트가 가장 먼저 하는 일은 내 파일시스템을 탐색하는 것이다. 6개월 전 OCR 벤치마크 레포를 만들어뒀다면, 그 프로젝트를 다시 열어 이번 실험을 확장하는 형태로 진행한다. 관련 프로젝트가 없다면 새로 만든다. 이어서 서브에이전트 팀이 각자 격리된 세션에서 일을 나눠 맡는다. 한 에이전트는 의존성을 설치하고, 다른 에이전트는 glm-ocr, olm-ocr, pymupdf, tesseract, opendataloader 같은 도구들을 실행해 성능 지표를 수집한다. 실험이 끝나면 결과 보고서가 생성되고, 다시 Telegram으로 나에게 온다.

이것이 **두 번째 HITL 체크포인트**다. 나는 결과를 보고 "한국어 텍스트로도 해봐", "GPU 타이밍도 측정해줘" 같은 추가 지시를 내리거나, 결과가 충분하다고 판단하면 다음 단계로 넘어간다.

**3단계 — 블로그 초안 작성**

결과가 만족스러우면 "블로그 글 써줘"라고 한다. 라이팅 에이전트가 별도 세션에서 실험 결과를 바탕으로 블로그 초안을 작성한다. 초안이 Telegram으로 오면 내가 읽고 피드백을 준다. "서론이 너무 길어", "코드 예제 추가해줘" 같은 식으로. 에이전트가 수정하고 다시 보낸다.

이것이 **세 번째 HITL 체크포인트**다. 초안 검토와 수정 루프가 여기서 일어난다.

**4단계 — 퍼블리싱**

내가 최종 승인하면 퍼블리셔 에이전트가 별도 세션에서 깨어나 내 블로그에 글을 자동 업로드한다.

이것이 **네 번째 HITL 체크포인트**이자 마지막이다. 내 승인 없이는 아무것도 공개되지 않는다.

### 1.3 핵심 아키텍처 제약: 세션 격리

이 시스템에서 가장 중요한 설계 원칙은 **모든 태스크가 완전히 별도의 에이전트 세션에서 실행된다**는 것이다. 모니터링 에이전트, 연구 에이전트, 라이팅 에이전트, 퍼블리셔 에이전트는 서로 메모리와 상태를 공유하지 않는다. 오케스트레이터가 이들을 조율하지만, 각 서브에이전트는 자신에게 주어진 컨텍스트만 알고 동작한다. 이 격리 원칙은 두 가지 이유에서 중요하다. 첫째, 한 에이전트가 잘못된 판단을 해도 다른 에이전트에게 영향이 퍼지지 않는다. 둘째, 각 에이전트의 컨텍스트 윈도우를 깨끗하게 유지해 성능 저하를 막는다.

## 2. 권장 셋업 구성

### 2.1 보유 디바이스와 역할 배분

보유한 장비는 세 가지다. OCI ARM Free Tier 인스턴스, MacBook Pro(메인 개발 머신), MacBook Air 2017(인텔, 서브 머신)이다. 각 장비에 어떤 역할을 맡길지, 그리고 왜 그렇게 배분하는지가 설계에서 핵심이다.

**OCI ARM Free Tier**는 이 시스템의 게이트웨이이자 실험 실행 환경이다. Ubuntu, 4 OCPU, 24GB RAM, 200GB 스토리지를 무료로 제공한다. 항상 켜져 있고, 개인 데이터가 없으며, 필요하면 언제든 초기화해서 다시 만들 수 있는 깨끗한 환경이다. OpenClaw Gateway와 실험 에이전트들이 여기서 돌아간다. NemoClaw는 설치하지 않는다. 이유는 뒤에서 자세히 설명하겠지만, OCI는 개인 파일이 없는 일회용 환경이기 때문에 Docker 컨테이너 격리만으로 충분하다.

**MacBook Pro**는 OpenClaw Node와 NemoClaw가 함께 설치된다. 이곳에는 수년간 쌓인 개인 파일, SSH 키, 환경 변수, 각종 자격증명, 개발 환경이 존재한다. 에이전트가 이 머신에서 코드를 실행할 때는 훨씬 강력한 격리가 필요하다. NemoClaw의 4-레이어 OS 격리(Docker + Landlock MAC + seccomp + 네트워크 네임스페이스)가 정당화되는 이유가 바로 여기에 있다. NemoClaw는 OpenClaw의 플러그인으로 작동하므로 OpenClaw Node와 함께 설치된다.

**MacBook Air 2017(인텔)**은 선택 사항이다. 오래된 하드웨이라 연산 능력은 제한적이지만, OpenClaw Node 소프트웨어 자체는 구동 가능하다. 포함한다면 경량 모니터링 보조 노드 정도의 역할만 맡기는 것이 적절하다. 실험 실행이나 LLM 추론은 맡기지 않는다.

### 2.2 전체 아키텍처 다이어그램

```
[OCI ARM — Gateway / 실험 환경]
OpenClaw Gateway (오케스트레이터)
├── Monitoring Agent  ← 별도 세션, 6시간마다 스케줄
├── Research Agent    ← 별도 세션, 온디맨드
├── Writing Agent     ← 별도 세션, 온디맨드
└── Publisher Agent   ← 별도 세션, 온디맨드
         ↕ Telegram (아웃바운드 + 인바운드)

Tailscale VPN Mesh
    ├── MacBook Pro: OpenClaw Node + NemoClaw
    └── MacBook Air 2017: OpenClaw Node (경량, 선택사항)
```

세 노드는 Tailscale VPN 메시로 연결된다. OCI에서 MacBook Pro로, 혹은 MacBook Pro에서 OCI로 에이전트가 이동하거나 파일을 참조할 때 이 VPN 터널을 통한다. Tailscale은 NAT 뒤의 MacBook도 문제없이 연결하며, OCI와 macOS 플랫폼 모두 공식 지원한다.

### 2.3 왜 프레임워크를 섞어 쓸 수 없는가

여기서 흔히 드는 의문이 있다. "MacBook Pro에는 IronClaw처럼 더 강력한 보안 프레임워크를 쓰고, MacBook Air에는 PicoClaw 같은 경량 프레임워크를 쓰면 안 될까?" 결론부터 말하면 안 된다. 프레임워크 믹싱은 이 아키텍처에서 작동하지 않는다.

이유는 프로토콜 비호환성이다. OpenClaw의 Gateway-Node 아키텍처는 노드 간 통신에 ACP(Agent Communication Protocol)를 사용한다. IronClaw, PicoClaw, NanoClaw, TinyClaw는 각각 자체 내부 통신 프로토콜을 갖고 있으며, ACP를 이해하지 못한다. OpenClaw Gateway가 "Node야, 이 태스크를 실행해"라고 ACP 메시지를 보내도, IronClaw 인스턴스는 그 메시지를 받아서 처리할 수 없다. 이것은 구현의 문제가 아니라 설계의 문제다. 각 프레임워크는 자신의 생태계 안에서 독립 실행형으로 설계되어 있다. 누군가의 Node가 되도록 설계된 것은 OpenClaw Node뿐이다.

따라서 OpenClaw Gateway를 쓴다면 모든 노드는 OpenClaw Node여야 한다. 그 위에 플러그인(NemoClaw 등)을 얹는 것은 가능하지만, 노드 자체를 다른 프레임워크로 교체할 수는 없다.

### 2.4 HITL 채널: Telegram

에이전트와 나 사이의 통신 채널은 Telegram이다. 중요한 것은 방향이다. 이 시스템에서 Telegram은 내가 에이전트를 조작하는 채널이 아니라, 에이전트가 나에게 먼저 연락하는 채널이다. 물론 양방향이다. 에이전트가 메시지를 보내면 나도 같은 Telegram 채팅에서 답한다. OpenClaw는 이 아웃바운드 Telegram 메시징을 기본 기능으로 내장하고 있다.

## 3. 선택 이유 — 왜 OpenClaw인가

### 3.1 분석한 13개+ 프레임워크 중 유일한 선택지

이 저장소에서 분석한 프레임워크가 13개가 넘는다. OpenClaw, Nanobot, NanoClaw, IronClaw, ZeroClaw, PicoClaw, TinyClaw, OpenFang, OpenJarvis, NemoClaw, NullClaw, Hermes Agent, Claude Code 등이다. 그중에서 내가 원하는 워크플로우를 처음부터 지원하는 프레임워크는 OpenClaw 하나뿐이다. 이것은 OpenClaw가 "제일 좋은 프레임워크"라는 뜻이 아니다. 내가 원하는 구체적인 기능들의 교집합이 OpenClaw에만 존재한다는 뜻이다.

그 기능들을 하나씩 살펴보자.

**첫째, R32 Gateway-Node 패턴이다.** 분석한 모든 프레임워크 중 공식 멀티 디바이스 Gateway-Node 아키텍처를 문서화한 것은 OpenClaw뿐이다. OpenJarvis, Hermes Agent, NullClaw는 모두 단일 머신 프레임워크로 설계되어 있다. OCI + MacBook Pro + MacBook Air를 하나의 에이전트 메시로 묶으려면 이 패턴이 필수다. OpenClaw 공식 문서는 Oracle Cloud와 macOS에서 Tailscale VPN을 사용한 멀티 노드 구성을 명시적으로 다룬다. 다른 프레임워크는 이런 설정 자체를 상정하지 않았다.

**둘째, Telegram 아웃바운드 HITL이다.** 사용자가 에이전트를 쿼리하는 것과 에이전트가 사용자에게 먼저 알리는 것은 전혀 다른 기능이다. Claude Code는 터미널에서 사용자 명령을 기다린다. OpenJarvis는 웹훅이나 API 폴링으로 메시지를 받는다. 이 둘은 에이전트가 먼저 연락하는 구조가 아니다. OpenClaw는 Telegram을 12개 메시징 채널 중 하나로 공식 지원하며, 에이전트 코드에서 Telegram으로 메시지를 발송하는 것이 기본 기능이다. 커스텀 코드를 짜지 않아도 된다.

**셋째, 내장 스케줄러다.** 24/7 모니터링 에이전트는 주기적으로 깨어나야 한다. 6시간마다 커뮤니티를 순회하고, 흥미로운 것이 없으면 다시 잠드는 것이 기본 행동 패턴이다. OpenClaw는 Cron 자동화를 내장 기능으로 제공한다. Claude Code에는 스케줄러가 없다. 별도의 cron 잡을 운영체제 레벨에서 관리하고, 그것이 Claude Code를 깨우고, Claude Code가 작업하고, 결과를 어딘가에 저장하고, 다음번 실행을 위해 상태를 보존하는 복잡한 글루 코드를 직접 작성해야 한다. 사실상 OpenClaw의 일부를 처음부터 다시 만드는 셈이다.

**넷째, `spawn_agent()`와 세션 격리다.** 내 시스템의 핵심 설계 원칙이 모든 태스크를 별도 세션에서 실행하는 것이라고 앞에서 설명했다. OpenClaw의 `spawn_agent()`는 진짜로 격리된 세션을 생성한다. 각 서브에이전트는 자신만의 메모리와 컨텍스트를 갖고, 오케스트레이터는 상태를 공유하지 않으면서 이들을 조율한다. TinyClaw도 멀티에이전트 팀을 지원하지만 단일 머신 내 체인 실행이다. 진정한 세션 격리와 멀티 디바이스 분산 실행은 OpenClaw의 고유한 영역이다.

**다섯째, Tailscale 공식 통합이다.** 멀티 디바이스 설정에서 VPN은 필수다. OCI 인스턴스와 NAT 뒤의 MacBook을 안정적으로 연결해야 한다. OpenClaw 공식 플랫폼 가이드는 Tailscale을 OCI + macOS 토폴로지에서 명시적으로 다룬다. 다른 프레임워크는 이런 네트워크 설정 자체를 문서에서 다루지 않는다.

### 3.2 대안들이 부적합한 이유

각 대안이 왜 이 시나리오에 맞지 않는지 구체적으로 짚어보자.

**Claude Code**는 강력한 도구이지만 이 시나리오에는 근본적으로 맞지 않는다. 스케줄러가 없고, Telegram 아웃바운드가 없고, Gateway-Node 아키텍처가 없다. Claude Code는 개발자가 터미널에서 명시적으로 호출하는 도구로 설계되어 있다. 24/7 자율 모니터링과 아웃바운드 알림을 Claude Code 위에 구현하려면 결국 OpenClaw가 이미 제공하는 것들을 처음부터 다시 만들어야 한다.

**OpenJarvis**는 스케줄러를 갖추고 있고 로컬 퍼스트 설계라 개인 사용자에게 어필하는 면이 있다. 그러나 멀티 디바이스 Gateway-Node 설계가 없다. 단일 머신 프레임워크다. OCI와 MacBook을 하나의 에이전트 메시로 엮으려면 OpenJarvis로는 불가능하다.

**Hermes Agent**는 Nous Research가 만든 Python 기반 프레임워크로, Claw 생태계와 직접 연결된 흥미로운 프로젝트다. 하지만 위임 아키텍처에 구조적 제한이 있다. `MAX_DEPTH=2`, `MAX_CONCURRENT=3`이다. 모니터링 → 연구(서브에이전트 팀) → 라이팅 → 퍼블리싱으로 이어지는 파이프라인은 이 제한에 금세 부딪힌다. 연구 단계에서 여러 서브에이전트가 병렬 실행되는 순간 `MAX_CONCURRENT=3`이 병목이 된다. 그리고 Hermes Agent에도 Gateway-Node 아키텍처는 없다.

**IronClaw + PicoClaw 믹싱**은 앞에서 설명했듯 프로토콜 비호환으로 불가능하다. IronClaw의 보안 모델은 인상적이다. WASM 샌드박스와 Zero-Exposure 크레덴셜 관리는 분석한 프레임워크 중 가장 강력하다. 하지만 IronClaw는 독립 실행형으로 설계된 프레임워크이며, OpenClaw Gateway의 Node로 동작할 수 없다. 보안이 좋다고 해서 아키텍처 호환성 문제가 해결되지는 않는다.

### 3.3 OCI Free Tier는 충분한가

OCI ARM Free Tier(4 OCPU, 24GB RAM)가 이 워크플로우에 충분한가? 결론부터 말하면 충분하다. 이 판단이 틀릴 것 같다면, 그것은 컴퓨팅 요구사항과 LLM API 요구사항을 혼동하기 때문이다.

이 시스템의 실제 컴퓨팅 부하를 생각해보자. 모니터링 에이전트는 웹 크롤링과 API 호출을 한다. 연구 에이전트는 tesseract, pymupdf, opendataloader 같은 CPU 기반 도구들을 실행한다. 라이팅 에이전트는 텍스트를 생성한다. 이 모든 작업에서 무거운 연산은 LLM API 호출이고, 그 연산은 Anthropic의 클라우드 서버에서 일어난다. OCI 인스턴스가 하는 일은 프롬프트를 보내고 응답을 받아서 처리하는 것이다. 여러 Docker 컨테이너를 동시에 실행하는 것도 24GB RAM에서 충분히 가능하다.

진짜 비용 드라이버는 Claude API 토큰 사용량이다. 서버 사양이 아니다. 즉, OCI Free Tier를 쓴다고 해서 시스템 성능이 저하되지 않는다. 더 큰 인스턴스로 업그레이드해도 Claude API 응답 속도는 동일하다. OCI Free Tier는 이 워크플로우에 완전히 적합하다.

### 3.4 NemoClaw를 MacBook Pro에만 설치하는 이유

NemoClaw를 OCI에 설치하지 않고 MacBook Pro에만 설치하는 결정은, 두 환경이 근본적으로 다른 위험 프로파일을 갖고 있기 때문이다.

OCI 인스턴스는 처음부터 깨끗하게 만든 전용 클라우드 환경이다. 개인 파일이 없고, SSH 키가 없고(OCI 관리형 키 사용), 수년간 쌓인 개발 환경이 없다. 에이전트가 실험을 실행하다가 환경이 오염되면 인스턴스를 초기화하고 다시 만들면 된다. 이것이 클라우드 인스턴스의 본질적 특성이다. 이런 환경에서는 Docker 컨테이너 격리만으로 충분하다. 컨테이너가 뚫려도 잃을 것이 없다.

MacBook Pro는 완전히 다르다. 이 머신에는 수년간 축적된 SSH 키, GitHub 토큰, AWS 크레덴셜, 개발 환경 설정, 개인 코드베이스, 개인 문서가 있다. 에이전트가 이 머신에서 코드를 실행할 때, 그 코드가 `/Users/jaesolshin` 디렉토리를 마음대로 읽거나 네트워크 요청을 외부로 보낼 수 있다면 심각한 문제다. 바로 여기서 NemoClaw의 4-레이어 격리가 의미를 가진다.

NemoClaw의 4-레이어를 구체적으로 살펴보면: Docker 컨테이너는 기본 프로세스 격리를 제공한다. Landlock MAC(Mandatory Access Control)은 Linux 커널 수준에서 파일시스템 접근을 제한한다 — 에이전트는 `/sandbox` 디렉토리만 읽고 쓸 수 있다. seccomp은 시스템 콜을 필터링해 권한 상승을 막는다. 네트워크 네임스페이스는 컨테이너의 외부 네트워크 접근을 OpenShell 게이트웨이를 통해서만 허용한다. API 키는 컨테이너 안에서 절대 직접 접근할 수 없고, OpenShell이 주입한다.

NemoClaw는 OpenClaw의 플러그인으로 설계되었으므로(독립 실행형 프레임워크가 아님) MacBook Pro의 OpenClaw Node 위에 자연스럽게 얹힌다. 프로토콜 호환성 문제가 없다.

### 3.5 Andrej Karpathy와 Nanobot에 대한 정확한 사실

연구 과정에서 "Andrej Karpathy가 Nanobot을 추천했다"는 이야기를 접했을 수 있다. 이 내용을 정확히 정정해둘 필요가 있다. Nanobot 개발자(@huang_chao4969, HKU-Data Intelligence Lab 소속 — OpenClaw 생태계 연구에서 DeepInnovator를 만든 같은 연구소)가 X에 올린 내용은 Karpathy가 자신들의 프로젝트를 "noticed(주목했다)"는 것이다. 주목과 추천은 다르다. Karpathy가 프로젝트를 언급했다고 해서 그것이 이 시나리오에 적합하다는 뜻은 아니다.

Nanobot 자체를 보면, 총 코드베이스가 약 4,000줄에 불과한 연구용 초경량 에이전트다. OpenClaw의 400,000줄 코드베이스와 비교하면 의도적으로 최소주의를 추구한다. 연구자가 에이전트 동작을 완전히 이해하고 수정하기 쉽도록 설계된 것이다. MCP를 지원하고 Python으로 확장 가능하다는 장점이 있지만, 멀티 디바이스 Gateway-Node 아키텍처, 스케줄러, Telegram 아웃바운드 HITL을 지원하지 않는다. 내가 설계하는 시스템에는 적합하지 않다.

## 4. 다음 단계

### 4.1 실행 순서

설계가 확정되었으니 실제로 구축하는 순서를 정한다. 각 단계는 다음 단계의 전제 조건이므로 순서대로 진행해야 한다.

**1단계: OCI에서 OpenClaw Gateway 설치 및 초기화**

가장 먼저 OCI ARM 인스턴스에 OpenClaw를 설치한다. 이 인스턴스가 전체 시스템의 두뇌 역할을 하므로 가장 먼저 올라와야 한다. 설치 후 Gateway 모드로 초기화하고, 관리자 계정을 설정한다. 이 시점에서 OpenClaw 웹 UI가 로컬호스트에서 접근 가능한지 확인한다.

**2단계: Tailscale 설치 및 OCI ↔ MacBook Pro 메시 구성**

OCI 인스턴스와 MacBook Pro 모두에 Tailscale을 설치하고 같은 Tailscale 계정으로 연결한다. Tailscale이 할당하는 사설 IP(100.x.x.x 대역)로 두 노드가 서로 핑이 통하는지 확인한다. 이 VPN 메시가 노드 간 ACP 통신의 기반이다.

**3단계: Telegram Bot 토큰 발급 및 OpenClaw Telegram 플러그인 연동**

Telegram BotFather에서 새 봇을 만들고 토큰을 발급받는다. OpenClaw의 Telegram 플러그인 설정에 이 토큰을 입력한다. 테스트로 에이전트가 Telegram 채팅으로 메시지를 보내는지, 내가 답하면 에이전트가 받는지 양방향을 확인한다. 이 확인 없이 다음 단계로 넘어가면 안 된다.

**4단계: MacBook Pro에 OpenClaw Node + NemoClaw 설치**

MacBook Pro에 OpenClaw Node를 설치하고 OCI Gateway에 등록한다. Tailscale IP를 사용해 연결한다. 이어서 NemoClaw를 OpenClaw 플러그인으로 설치한다. NemoClaw 온보딩 과정에서 Docker 설치 여부와 GPU 환경을 자동으로 감지한다. MacBook Pro에 GPU가 없다면(Intel Mac 또는 해당 모델이 외장 GPU 없는 경우) NemoClaw는 CPU 전용 프로파일로 설정된다.

**5단계: Monitoring Agent 스케줄 설정 및 Telegram 연동 테스트**

OpenClaw의 Cron 기능을 사용해 모니터링 에이전트를 6시간마다 실행하도록 스케줄을 설정한다. 첫 실행에서 에이전트가 실제로 깨어나서 커뮤니티를 탐색하고, 결과를 판단하며, Telegram 메시지를 전송하는지 전체 흐름을 확인한다. 이 엔드투엔드 테스트가 통과하면 시스템의 핵심 루프가 작동하는 것이다.

### 4.2 이 설계의 확장 가능성

일단 이 다섯 단계가 완료되면 시스템의 뼈대가 완성된다. 이후는 에이전트의 판단 품질을 개선하는 작업이다. 모니터링 에이전트가 "흥미롭다"고 판단하는 기준을 정교하게 만들고, 연구 에이전트의 실험 설계를 개선하고, 라이팅 에이전트의 문체를 내 블로그 스타일에 맞게 조정하는 것들이다. 이런 개선은 에이전트 프롬프트와 스킬 설정의 영역이지, 아키텍처를 바꾸는 작업이 아니다. 기반을 올바르게 구축했다면 이후의 개선은 훨씬 쉬워진다.
