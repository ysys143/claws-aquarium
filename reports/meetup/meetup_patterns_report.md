## 1. 2-Step Cron 최적화 패턴

**소스**: 김동규 발표 (12_김동규_nanoclaw_사용기)

### 문제

기존 크론잡 방식: 주기마다 에이전트를 완전히 깨워 작업 수행 → 실제로 할 일이 없어도 LLM 호출 비용 발생.

### 해결책: 2단계 레이어

```
[1단계: 경량 조건 체크]
  JavaScript/Bash 코드 스니펫
  "새 GitHub 이슈가 있는가?" (API 체크)
  → false: 에이전트 호출 없이 종료 (토큰 0)
  → true: 2단계 진행

[2단계: 조건부 에이전트 스폰]
  조건 충족 시에만 에이전트 깨우기
  실제 작업 수행
```

### 적용 사례

- 15분 주기 GitHub 이슈 체크: 새 이슈 없으면 에이전트 호출 없음
- SNS 트렌드 모니터링: 트렌드 변화 없으면 스킵
- 빌드 상태 체크: 실패 시에만 알림 에이전트 호출

### 효과

| 방식 | 토큰 소비 (일 96회 체크 시) | 실제 작업 비율 |
|------|----------------------|-------------|
| 기존 (항상 에이전트 호출) | 96회 × LLM 비용 | 5~10% |
| 2-Step 패턴 | 5~10회 × LLM 비용 | 100% |

## 2. Subagent Context 격리 패턴

**소스**: 윤주운 발표 (09_윤주운_Arbiter_FLOCK_워크플로우, Sionic AI)

### 문제

OpenClaw 서브에이전트는 부모 에이전트의 모델·메모리·시스템 프롬프트를 모두 상속 → 도메인별 메모리 격리 불가 → 컨텍스트 오염 발생.

**예**: 퀀트 트레이딩 에이전트가 마케팅 에이전트의 컨텍스트를 상속받아 잘못된 판단.

### 해결책: 독립 에이전트 + 메시지 채널 연결

```
[기존 서브에이전트]
  부모 에이전트 → 서브에이전트 (모든 컨텍스트 상속) → 오염

[격리 패턴]
  에이전트 A (도메인 A 메모리만) ─┐
  에이전트 B (도메인 B 메모리만) ─┼─ Mattermost/Slack 채널
  에이전트 C (시스템 관리만)     ─┘
```

### FLOCK 시스템 실전 구현

- **Mac Mini**: OpenClaw 기반 오케스트레이터 노드
- **워크스테이션 (RTX 5090 × 2)**: AI 리서치, 로컬 모델 실험
- **데스크탑 (RTX 3090 × 2)**: 로컬 모델 서빙
- **Synology NAS**: 데이터셋·모델 저장, Mattermost 서버
- **연결**: Tailscale VPN + SSH

### 장기 암묵지 관리

- 과거 Claude Code 대화에서 시행착오·노하우 자동 추출
- 임베딩 벡터로 저장
- 새 에이전트에 주입 → 반복 실수 방지
- Claude = PM 역할 (코딩은 Codex에 위임)

## 3. FSM 결정론적 에이전트 패턴

**소스**: 한수관 발표 (11_한수관_ResearchClaw, AER Labs)

### 문제

단일 에이전트에 모든 것을 맡기면:
- 컨텍스트가 뒤섞임
- 플래닝·실험·평가 코드 혼재
- 결정론적 제어 없으면 연구 흐름을 에이전트가 제대로 따르지 못함

### 해결책: FSM(유한 상태 기계) 기반 단계 분리

```
[Planning 단계]
  사람과 상호작용
  요구 스펙 → 구현 스펙 단계별 명세화
  디렉토리: /planning/

[Experiment Loop 단계]
  코드 작성 후 반드시 실행되도록 강제하는 툴콜
  디렉토리: /experiment/
  권한: /experiment/ 내에서만 쓰기 가능

[Evaluation 단계]
  시각화 시스템 (별도 설계 필요 - 병목 지점)
  디렉토리: /evaluation/
  권한: 읽기만
```

### 디렉토리 레벨 권한 격리

- 각 에이전트는 자신의 단계 디렉토리에서만 작업
- Planning 에이전트 → /experiment/ 쓰기 불가
- Experiment 에이전트 → /evaluation/ 수정 불가
- 안정성의 핵심: "각 에이전트에게 특정 디렉토리 내에서만 작업하도록 권한을 제한"

### 적용 결과 (ResearchClaw)

- 반도체 RTL 코드 자동 생성 확인
- 로보틱스 픽앤플레이스 정상 동작
- 시뮬레이션 환경에서 안정적 동작 확인
- 실제 하드웨어 전환 시 추가 챌린지 존재 (레이턴시, 툴콜 해석 오류)

## 4. 3단계 HITL 승인 워크플로우

**소스**: 정세민 발표 (16_정세민_Ultraworker_AI_Orchestration, Sionic AI), 최재훈 발표 (03_최재훈)

### Ultraworker 4단계 워크플로우 (정세민)

```
[1단계] 컨텍스트 탐색 (explore context 스킬)
    ↓ [승인 1]
[2단계] 작업 리스트 작성 + 슬랙 BlockKit 승인 요청
    ↓ [승인 2]
[3단계] 테크 스펙 작성 (복잡한 업무만)
    ↓ [승인 3]
[4단계] 구현 완료 후 보고
```

**슬랙 BlockKit UI**: 좋아요 버튼으로 각 단계 승인/거부. 잘못된 명령 시 에이전트 강제 종료(terminate) 가능.

**세션 포크**: Claude Code 세션을 포크하여 맥락을 이어받아 다음 세션으로 연속 진행.

### OpenClaw 3-Layer Approval (최재훈)

```
[Layer 1: 디바이스 승인]
  어떤 기기에 명령을 내릴 것인가?

[Layer 2: 노드 승인]
  해당 기기의 어떤 노드(기능 서버)를 사용할 것인가?

[Layer 3: Capability 승인]
  해당 노드의 어떤 기능을 실행할 것인가?
```

**보안 의의**: 각 레이어에서 별도 승인 → 에이전트가 허용 범위를 넘어서는 행동 원천 차단.

## 5. Fork = Product 패턴

**소스**: 김동규 발표 (12_김동규_nanoclaw_사용기)

### 개념

기존 오픈소스: 기능을 PR로 제출 → 메인테이너가 머지 → 모두가 동일 버전 사용.

**Fork = Product 패턴**: 코드베이스가 충분히 작고 이해 가능하면, 각 사용자가 직접 포크하여 자신에게 맞게 커스터마이즈 = 포크 자체가 제품.

### nanoclaw 사례

- 핵심 코드베이스 약 4,000줄 (인간과 AI 에이전트 모두 머릿속에 담을 수 있는 규모)
- 외부 PR 미수용
- 사용자가 직접 포크하여 Slack 연동, 크론잡 최적화, 공유 지식 저장소 등 구현
- Andrej Karpathy가 "AI 에이전트의 머릿속에도 들어올 정도의 코드베이스"로 칭찬

### 에이전트 네이티브 오픈소스의 새 포맷

| 기존 오픈소스 | Fork = Product |
|-------------|----------------|
| 모든 기능 중앙화 | 핵심만 유지 |
| PR로 기능 추가 | 포크로 커스터마이즈 |
| 대규모 코드베이스 | ~4,000줄 (에이전트도 이해 가능) |
| 안정성 우선 | 이해 가능성 우선 |

| 역할 | 담당 | 강점 |
|------|------|------|

### 정우석 4단계 스킬 개발 프로세스

```
[1단계: 요청]
  "URL과 함께 이 기능을 어떻게 실행하는지, 어떤 우려가 있는지 질문"

  계획 수립 → 반복 피드백 → 실행

  트리거 워드 + 문서화 요청 → 스킬 완성

## 7. Chat Template 직렬화 패턴

**소스**: 민용기 발표 (08_민용기_에이전트_엔진룸_툴콜링, FriendliAI)

### 핵심 원리

LLM은 실제로 텍스트 시퀀스만 처리 → JSON 도구 정의도 텍스트로 직렬화됨 → 이 직렬화 방식이 모델마다 다름 → **Chat Template이 성능에 직접 영향**.

### 모델별 Tool Call 포맷 비교

| 모델 | 내부 포맷 특성 |
|------|-------------|
| **GLM 4.7** | 자체 포맷 (중국 모델) |
| **Gemini** | 별도 직렬화 방식 |
| **GPT** | OpenAI JSON 스키마 |
| **Claude** | Anthropic XML/JSON 혼합 |

**핵심 발견**: 같은 모델이라도 채 템플릿 튜닝만으로 성능 차이 발생.

### BFCL 7~8% 성능 향상 실험

- **대상**: Gemini 2.5 Flash
- **방법**: 자체 툴 콜 파서 + 프롬프트 인젝터 적용
- **결과**: BFCL(Berkeley Function Calling Leaderboard) 싱글/패럴렐 배치 평가에서 약 7~8% 성능 향상
- **의미**: API 레이어를 넘어 직렬화 텍스트 레이어까지 최적화하면 유의미한 성능 개선 가능

### 실무 적용 지침

- 로컬 모델 사용 시 기본 채 템플릿이 최적이 아닐 수 있음
- BFCL 등 공개 벤치마크로 튜닝 효과 검증 권장
- 인퍼런스 엔진(vLLM, SGLang)별로 파싱 구현 상이함 주의

## 8. HashLine 편집 패턴

**소스**: 민용기 발표 (08_민용기)

### 문제: 기존 String Replace 방식의 한계

```
old_string: "function foo() {\n  return bar;\n}"
new_string: "function foo() {\n  return baz;\n}"
```

- 문자열이 파일 내 다른 곳에도 존재하면 오류
- 모델이 정확한 old_string을 재현하기 어려움
- 특히 로컬 모델의 컨텍스트 윈도우 제한 환경에서 실패율 높음

### 해결책: HashLine 편집

각 라인에 해시 앵커를 부여 → 모델이 변경 대상 라인만 해시로 지정:

```
# 파일 내용 (해시 앵커 포함)
#a3f2 function foo() {
#b7c1   return bar;   ← 변경 대상
#d4e9 }

# 편집 명령
CHANGE #b7c1 TO "  return baz;"
```

### String Replace vs HashLine 비교

| 항목 | String Replace | HashLine |
|------|---------------|---------|
| **정확도** | 문맥 의존 | 라인 고유 식별 |
| **실패 원인** | 중복 문자열 | 해시 충돌 (드묾) |
| **로컬 모델 적합성** | 낮음 | 높음 |
| **컨텍스트 효율** | old+new 모두 필요 | 해시만 필요 |
| **구현 복잡도** | 낮음 | 중간 |

## 10. 세컨드 브레인 + AI 에이전트 통합 패턴

- **AI 담당**: Capture, Organize, Distil (수집·정리·증류)
- **사람 담당**: 무엇을 저장·버릴지 판단 + 방향 설정 + Express

### 실전 시스템 아키텍처

```

[동기화] Nextcloud + WebDAV (멀티 디바이스)
    ↓ 주기적
[백업] GitHub (마크다운 diff 이력 추적)

## 결론

meetup 실전 패턴들은 프레임워크 코드베이스 분석에서는 보이지 않는 **운영 레이어의 지혜**를 담고 있다.

핵심 인사이트:
1. **비용 최적화**: 2-Step Cron만으로 LLM 비용을 90% 절감 가능
2. **안정성**: FSM + 디렉토리 권한 격리 = 결정론적 에이전트 제어
3. **확장성**: Subagent 격리 = 전문화된 에이전트 팀 운영
4. **사용성**: Chat Template 최적화 = API 비용 없이 7~8% 성능 향상
5. **민주화**: 17만원 미니 PC로도 실용적인 AI 세컨드 브레인 구축 가능
