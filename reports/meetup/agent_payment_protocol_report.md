## 개요

AI 에이전트가 경제 활동의 주체가 되려면, 인간 간 거래를 위해 설계된 기존 결제·신원 시스템과 구조적으로 불일치한다. 이 보고서는 에이전트 네이티브 결제·신원·상호운용성 프로토콜을 비교 분석한다.

**기존 결제 방식의 근본 문제**:

| 방식 | 문제점 |
|------|--------|
| API 키 | 인증만 가능, 에이전트 간 자동 정산 불가 |
| 구독/크레딧 | 작업 단위 거래를 다른 에이전트에게 분배 어려움 |
| PG 결제 (카드/카카오페이) | 인간이 직접 승인해야 함 → 에이전트가 대기 |

**필요 조건 (머신 네이티브 페이먼트)**:
1. 퍼미션리스(Permissionless): 사전 승인 불필요
2. 조건부 결제(에스크로): 작업 완료 후 정산
3. 확장성: 서로 다른 프로토콜 간 자연스러운 결합
4. 누가 작업 완료를 판단·검증할 것인가
5. 자금 흐름의 투명한 추적 (세금·회계)

## 1. X402 — HTTP 402 기반 결제 미들웨어

**소스**: Logan Kang 발표 (02_Logan_Kang_Composable_Agent_Economy, BASE Korea)

### 개념

HTTP 상태 코드 402 "Payment Required"에서 영감을 받은 결제 미들웨어. 백엔드 엔드포인트 앞에 붙여 사용.

### 동작 방식

```
[클라이언트 (에이전트)]
  → 유료 API 엔드포인트 호출

[서버]
  → 402 Payment Required 응답 (결제 조건 포함)

[클라이언트]
  → 블록체인 지갑으로 결제 서명(수표) 생성하여 재전송

[서버]
  → 서명 검증
  → 요청 처리 및 정산

※ 전체 흐름에 인간 개입 없음
※ 지갑에 예산 충전은 인간이 수행
```

### 특징

| 항목 | 내용 |
|------|------|
| **결제 방식** | 블록체인 지갑 서명 (수표 방식) |
| **다이나믹 프라이싱** | 지원 |
| **카드/은행 계좌** | 스펙상 지원, 현재는 블록체인만 실용적 |
| **인간 개입** | 없음 (지갑 충전만 인간 수행) |
| **레이어 스택** | UCP → AP2 → X402 순서로 쌓임 |

### 에이전트 커머스 레이어 스택

```
[UCP] 커머스 회사가 에이전트 전용 엔드포인트 제공
  ↓
[AP2 / A2A] 에이전트 간 통신 및 결제 절차
  ↓
[X402] HTTP 미들웨어 수준 결제 처리
  ↓
[Base/Ethereum] 블록체인 정산
```

## 2. Virtuals ACP — 에이전트 마켓플레이스 프로토콜

**소스**: Logan Kang 발표 (02_Logan)

### 개념

에이전트의 스킬·기능을 마켓플레이스에 등록하고 판매하는 프로토콜. AP2 역할 통합.

### 핵심 기능

1. **서비스 디스커버리**: 어떤 에이전트가 어떤 기능을 파는지 그래프 형태로 검색
2. **흥정(Negotiation)**: 에이전트 간 가격 협상 기능 (예: "0.1 → 0.09로 해줘") — 프로토콜 수준에서 내장
3. **에스크로 기반 거래**: 작업 대금을 스마트 컨트랙트에 잠금 → 작업 완료 검증 시 정산
4. **결과 평가(Evaluation)**: 납품물 자동 검증 로직 지원

### 개발자 구현 범위

```
[개발자가 구현할 것]
  - 에이전트 기능 (로직)
  - 평가 코드 (작업 완료 검증)
  - openclaw-acp 툴로 등록

[Virtuals ACP가 처리]
  - 서비스 발견
  - 가격 협상
  - 에스크로 생성/정산
  - 평판 누적
```

### 소액 결제 단위

- USDC 기준 10⁻⁶달러 (약 0.001원) 수준까지 지원
- 사소한 기능도 수익화 가능한 마이크로트랜잭션 기반 비즈니스 모델

### Base 블록체인의 역할

| 기능 | 설명 |
|------|------|
| 에스크로 | 작업 대금 임시 보관 (스마트 컨트랙트) |
| 투명성 | 거래 상태 온체인 기록 (누구나 조회) |
| USDC 지원 | 스테이블코인 기반 소액 반복 거래 |
| 불변성 | 스마트 컨트랙트 배포 후 수정 불가 = 코드 자체가 신뢰 |

## 3. ERC-8004 — 에이전트 신원 인증 표준

**소스**: 김서준 발표 (06_김서준_AI_에이전트와_온체인_경제, HASHED)

### 문제: 에이전트 신원 없음

현재 전 세계 수억 개의 에이전트가 활동하나, 신뢰를 확인할 수단 전무.

직원 채용 시 이력서·신분증을 확인하듯, 에이전트에게도 **KYD(Know Your Agent)** 과정 필요.

### 3-Layer 구조

```
[Layer 1: Registry]
  ERC-721 기반 에이전트 출생 신고 및 정보 등록
  "이 에이전트는 누가 만들었고 무슨 기능을 하는가"

[Layer 2: Reputation Registry]
  실제 블록체인 트랜잭션이 발생한 경우에만 리뷰 적립
  영수증 기반 평점 (조작 불가)
  "이 에이전트는 과거에 어떻게 일했는가"

[Layer 3: Validation]
  경량: 평판 확인만
  중간: 슬래싱 (자산 담보, 실수 시 손실)
  완전: 수학적·하드웨어 증명 (가장 강력)
```

### 참여 주체

MetaMask, Google, Ethereum Foundation, Coinbase 등이 공동 추진.

**블록체인 기반인 이유**: 특정 기업이 독점 불가한 중립적 인프라여야 전 세계 표준으로 채택 가능.

### 현재 한계

- 스마트 컨트랙트 취약점 존재
- 결제 범위 제한
- 개발자 경험(DX) 미흡
- 한국에서 에이전트 신원 인증 시도 팀 거의 없음

### 에이전트 경제 시나리오

```
개인 에이전트 (사용자)
  → 여행 에이전트
    → 이탈리아 현지 에이전트 (호텔 협상)
    → 항공사 에이전트 (항공권 예약)
  → 선금 결제 (X402)
  → 신뢰 근거 (ERC-8004 레퓨테이션)
```

신뢰 없이 이 흐름이 작동하려면 평판·결제 정보의 블록체인 기록이 필수.

## 4. AIP — 에이전트 상호운용성 프로파일 (Unibase)

**소스**: Zoe Chen 발표 (14_Zoe_Chen_Unibase_Extending_OpenClaw)

### 개념

에이전트 간 협력·조율을 위한 프로토콜. 두 가지 핵심 요소:

1. **신원(Identity)**: 어떤 에이전트인지, 어떤 능력을 보유하는지 식별
2. **결제(Payment)**: X402·EIP-4 프로토콜로 에이전트 간 직접 결제 (인간 개입 불필요)

### 미래 시나리오

```
[사용자가 여행 계획 요청]
  → 개인 에이전트 (Unibase 기억 보유)
    → AIP로 항공사 에이전트 발견
    → AIP로 호텔 에이전트 발견
    → X402로 예약금 자동 결제
    → 결과 통합 후 사용자에게 보고
```

### Unibase와 AIP의 관계

- **Unibase**: 크로스 디바이스 탈중앙화 메모리 레이어 + Beat Agent(토큰 런칭 플랫폼)
- **AIP**: Unibase 위에서 에이전트 간 협력·신원·결제를 처리하는 프로토콜 레이어

## 5. A2A — Google 에이전트 간 통신 표준

**소스**: 여러 발표 언급 (OpenFang, OpenJarvis 구현 확인)

### 개념

Google이 주도하는 에이전트 간 통신 표준. 에이전트끼리 교류하는 공용어.

### 현재 구현 상태

| 프레임워크 | A2A 지원 |
|----------|---------|
| OpenFang (9th) | 지원 (Google A2A 스펙 준수) |
| OpenJarvis (10th) | 지원 (Stanford/Scaling Intelligence Lab) |
| OpenClaw | 확인 안 됨 |
| 기타 Claw | 미지원 |

### A2A vs AIP 비교

| 항목 | A2A (Google) | AIP (Unibase) |
|------|-------------|--------------|
| **주체** | Google 주도 | Unibase |
| **범위** | 통신 표준 | 상호운용성 + 결제 + 신원 |
| **블록체인** | 없음 | 있음 (EIP-4) |
| **채택** | OpenFang, OpenJarvis | OpenClaw 스킬로 설치 |

## 프로토콜 통합 아키텍처

다섯 프로토콜은 계층적으로 쌓인다:

```
[Application Layer]
  에이전트 서비스 (OpenClaw 스킬, OpenJarvis, 등)

[Interoperability Layer]
  A2A (통신) | AIP (Unibase 협력)

[Identity Layer]
  ERC-8004 (신원·평판·검증)

[Commerce Layer]
  Virtuals ACP (서비스 발견·협상·에스크로·평가)

[Payment Layer]
  X402 (HTTP 402 미들웨어)

[Settlement Layer]
  Base / Ethereum (USDC 스테이블코인)
```

## 한국 생태계 현황

**김서준 (HASHED) 분석**:
- 한국 대기업·빅테크 중 API를 운영하는 회사가 손에 꼽을 정도
- 특정 플랫폼이 OpenAI 마켓플레이스에는 연결하면서 외부 개발자에게는 API 비공개
- 한국에서 에이전트 신원 인증을 시도하는 팀: 거의 없음 (발표 시점 기준)
- 한국 스테이블코인 법안 미통과 → 즉시 적용 어렵지만 준비 시작 권장

**B2A(Business to Agent) 비즈니스 모델 부상 전망**:
- 에이전트가 유저 컨텍스트를 가지고 쇼핑을 대행 → "에이전트 애즈(Agent Ads)"
- 구매 의도(intent)가 명확해 전환율이 매우 높아질 것으로 예측

## 결론 및 열린 질문

### 결론

에이전트 경제의 성립을 위한 인프라 4개 기둥:
1. **MCP** (기능 표현)
2. **A2A / AIP** (통신 및 협력)
3. **ERC-8004** (신원 인증)
4. **X402 / Virtuals ACP** (결제 및 에스크로)

이 중 MCP, A2A는 성숙 단계. **신원 인증(ERC-8004)과 머신 네이티브 결제(X402)가 현재 가장 미성숙하며 동시에 가장 중요한 미해결 영역**이다.

### 열린 질문

- ERC-8004 스마트 컨트랙트 취약점은 언제, 어떻게 해결될 것인가?
- 블록체인 기반 신원 vs 중앙화 신원(OAuth2 등)의 트레이드오프는?
- 한국 스테이블코인 법제화 이후 국내 에이전트 경제 생태계 형성 속도는?
- Virtuals ACP의 가격 협상 로직은 에이전트 담합을 어떻게 방지하는가?
- X402 다이나믹 프라이싱이 에이전트 간 가격 전쟁을 유발할 가능성은?
